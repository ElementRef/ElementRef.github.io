Electron 离屏渲染这一小节，是要表达什么的？看不懂意图

理解这一节的意图，核心在于明白一个看似矛盾的前提：

“既然 Electron 本身就是一个浏览器外壳，为什么我们要让它不显示任何窗口，而是在‘幕后’悄悄渲染网页？”

Electron 文档中引入“离屏渲染（Offscreen Rendering）”，并不是为了日常的桌面 App 界面开发，而是为了向你提供一种“把 Chromium 当作图形渲染引擎 / 数据源”的特权能力。

简单来说，它的核心意图是：“只用 Chromium 的渲染能力去画网页，但不要把它呈现在屏幕上。网页渲染出的每一帧画面，我们要直接拿走它的‘像素数据（Bitmap / 纹理）’去派其他用场。”

它的典型应用场景（能用来干什么？）

当你开启了 offscreen: true，Electron 就变成了一个没有肉身的“灵魂画手”。业界通常用它来做以下三件事：

1. 3D 游戏 / 虚拟世界中的“内嵌网页”（3D Texture）

比如你正在用 Unity、Unreal（虚幻引擎）或 Three.js 制作一个 3D 游戏。游戏中有一台 3D 电视机，你想让这台电视机播放一个真实的网页（比如 Bilibili 的弹幕视频，或者网页版聊天室）。

  • 做法： 在后台用 Electron 离屏渲染该网页。
  • 意图： 监听 paint 事件获取每一帧的图片（像素数据），然后把这张图片贴在 3D 场景里那个电视机的模型上（作为 3D 纹理）。
  • 效果： 玩家可以在 3D 游戏中实时看到并交互这个网页，而桌面上不会弹出一个多余的浏览器窗口。

2. 自动化截图与视频生成（Headless Node 工具）

你需要做一个后台服务，自动输入一个 URL，然后输出该网页的截屏，或者录制该网页的动态动画（比如一些在线生成 PPT、生成海报、生成视频的 SaaS 服务）。

  • 做法： 开启离屏渲染并加载目标页面。
  • 意图： 实时截获 paint 事件返回的 image（NativeImage 对象），将其转为 PNG 保存到硬盘，或者拼接成视频。

3. AR / VR 混合现实

在 AR / VR 设备中，传统的 2D 操作系统窗口是无法直接放置的。通过离屏渲染，你可以把整个网页界面变成三维空间中漂浮的一个画布，让用户戴着设备与网页交互。

  1. 没有窗口： BrowserWindow 依然存在，但它在桌面上是不可见且无实体的（Frameless & Invisible）。

  • Windows： 调用了 COM 组件的 SHAddToRecentDocs。

<webview> 和 <iframe> 有什么不同？为什么 Electron 不推荐 <webview>？

简单来说：<iframe> 是标准的 Web 技术，它的安全沙箱和控制权极其有限；而 <webview> 是 Chromium 早期的“黑科技”，它允许你在网页内嵌入一个近乎拥有“完全独立浏览器特权”的子窗口。

然而，由于<webview>底层的硬伤，Electron 官方现在不仅不推荐它，甚至在很多现代桌面应用中都在极力去它。

要理解这两者的本质区别和为什么它被嫌弃，我们需要从它们的底层架构和安全模型说起。

1. 核心技术对比：<webview> vs <iframe>

维度<iframe><webview>（Electron / Chrome Apps）
底层本质网页内的标准 HTML 节点。一个独立的、嵌入在当前渲染进程中的 Out-of-Process Frame（进程外子框架），本质上是由主进程直接管理的独立 webContents。
进程模型默认与父页面共享同一个渲染进程（除非触发跨域且开启了 Site Isolation）。绝对拥有独立的渲染进程，与你 App 外壳的渲染进程物理隔离。
特权与控制力受同源策略（SOP）和 CSP 的极严格限制。父页面几乎无法干涉跨域 iframe 内部的 DOM 或网络请求。父页面可以通过 Electron 的 API，直接拦截 <webview> 里的网络请求、注入自定义 Preload 脚本、控制前进后退，控制权极大。
安全沙箱相对安全，但如果与父页面同源，可能会被恶意脚本通过 window.parent 实施攻击。天然的沙箱隔离。外部网站运行在其中，绝无可能访问到父页面的 Node.js 特权 API。

2. 为什么 Electron 极力不推荐使用 <webview>？

既然 <webview> 拥有独立的进程、极强的控制力，而且还天然安全隔离，为什么 Electron 官方还在文档中挂起黄牌警告，极力劝阻开发者使用它呢？

这背后的原因主要有三个：

致命痛点一：底层的硬伤导致 Bug 频发

<webview> 最初是谷歌为了 Chrome Apps 生态设计的。由于它要在现有的渲染进程 DOM 树里，硬生生“抠出”一块区域给另一个独立的渲染进程去画图，这在 Chromium 底层涉及到了极其复杂的合成渲染（Compositing）机制。

  • 拖拽（Drag & Drop）与右键菜单 Bug： 在 <webview> 内部，点击右键、双击、或者尝试拖拽文件时，经常会发生事件丢失，导致右键菜单定位偏移或干脆无法弹出。
  • 焦点丢失： 当用户在 <webview> 里的输入框打字，然后点击你 App 外壳的输入框时，键盘焦点经常会卡死在 <webview> 里切不出来。
  • 官方无法修复： 这些 Bug 大多源于 Chromium 自身的架构设计，Electron 团队根本没有权限也无法从底层去修复它们。

致命痛点二：可怕的性能与内存开销

由于 <webview> 本质上是伪装成 HTML 节点的另一个 webContents：

  • 在页面里多写一个 <webview>，系统就会毫无节制地多开一个完整的 Chromium 渲染进程。
  • 每一个 <webview> 都会消耗几十到上百 MB 的内存。如果你做了一个“多标签页浏览器”功能，在前端用 Vue / React 渲染了 10 个 <webview> 节点，用户的电脑内存会瞬间被吃空。

致命痛点三：DOM 生命周期的隐患

在 React 或 Vue 中，如果页面发生重新渲染，DOM 节点可能会被无意中销毁并重建。 如果你的 <webview> 节点因为前端组件更新被重新挂载（Remount），它内部的所有状态（用户登录的 Session、表单输入、历史记录）会瞬间全部丢失，且无法恢复。这在前端工程中是一场灾难。

3. 2026 年，我们该用什么替代它们？

如果你现在需要开发一个在应用内加载第三方网页（如 GitHub、AI 聊天页面）的功能，官方和业界推荐的黄金方案是 WebContentsView（旧版本中称为 BrowserView）。

为什么 WebContentsView 是终极解法？

因为 WebContentsView 彻底摆脱了 DOM 树的束缚。

  1. 纯原生贴纸： 它是直接由主进程在你的窗口上方“贴”上去的一个原生 Chromium 视图。它不属于前端 HTML 的一部分，因此绝对不会受到前端 React / Vue 重新渲染、销毁 DOM 的影响。
  2. 没有事件 Bug： 因为它是原生的视图（View），所以右键、拖拽、焦点切换全部由 Chromium 原生接管，100% 灵敏，零 Bug。
  3. 内存可控： 它的生命周期和销毁完全由你在主进程（Node.js 端）手动控制，你可以非常优雅地缓存或销毁它，从而极大地节省内存。

「让 Dock 图标“弹跳”起来」竟然不是 macOS 自己实现的？

其实，它的确是 macOS 自己实现的，Electron 只是充当了那个“按下开关的传话筒”。

在 macOS 系统中，决定一个 Dock 图标什么时候弹跳、以什么频率弹跳、弹跳时带有什么物理曲线和动画，这套极其复杂的物理引擎和动画机制，100% 运行在操作系统的底层（由 macOS 的 Dock.app 进程和 Cocoa 框架掌控）。

那么，为什么我们还需要在 Electron 代码里写 app.dock.bounce() 呢？

1. 原生“弹跳”是一个受控的安全机制

macOS 的系统设计原则是：绝对不允许任何第三方 App 在没有经过系统 / 用户允许的情况下，在后台流氓式地、无节制地抢夺用户的视觉焦点。

如果操作系统的设计是“只要应用在后台收到新消息，系统就自动让它弹跳”，那用户的 Dock 栏在群聊轰炸时就会变成一个令人崩溃的“蹦迪现场”。

因此，macOS 把“弹跳”的控制权设计成了一个严格的、双向确认的 API 关卡：

  1. 开关在应用手里： 只有你（开发者）最清楚什么时候收到的消息是“十万火急”的（比如视频通话来电），什么时候是“无足轻重”的（比如群聊有人发了张表情包）。因此，系统把“触发弹跳的决定权”交给了应用的 API。
  2. 物理实现在系统手里： 当你在 Electron 里调用 app.dock.bounce('critical') 时，Electron 的 C++ 底层会向 macOS 系统发送一条高优先级的系统指令（调用了 Apple 的原生 API requestUserAttention:NSCriticalRequest）。
  3. 系统接管一切： 收到指令后，macOS 才会调动系统级的动画资源，让你的 App 图标在 Dock 栏上开始划出那道完美的物理抛物线。

2. 什么是“系统自动实现”的弹跳？

在 macOS 中，确实有一些弹跳是完全由系统默认触发，不需要你写一行代码的。比如：

  • 应用启动时： 当你双击打开任何一个 Electron 应用时，图标在 Dock 栏上会一上一下地弹跳，直到窗口完全加载出来。这个过程是 macOS 强制且自动实现的，目的是告诉用户：“别着急，App 正在后台努力加载中。”
  • 应用崩溃 / 无响应时： 如果你的 Electron 主进程卡死或者突然崩溃，系统有时也会让图标跳动，或者当你尝试在它卡死时点击它，它会跳动以示警告。

3. 为什么 Electron 专门提供这个 API？

在纯 Web 浏览器里，网页（SPA）运行在严格的安全沙箱中。你在网页里写任何 JavaScript 脚本，都绝对没有办法调用操作系统的 NSApp 接口。也就是说，网页绝对没办法让 Chrome 浏览器的图标在 Dock 栏弹跳。

而 Electron 的核心价值，就是打破了这条沙箱边界。

通过 app.dock.bounce()，Electron 在主进程（Node.js 端）搭了一座桥，让你可以像写原生 macOS 软件（用 Swift / Objective-C 编写的 App）一样，随时向系统发出“请用户注意”的申请。

总结

这就好比汽车里的“双闪警示灯”。闪烁的频率、电路的控制、灯泡的亮灭完全是汽车系统自己实现的；但什么时候遇到紧急情况、什么时候该按下那个红色的三角按钮，必须要由身为驾驶员（开发者）的你来决定。

Electron 内部的网页，最好是不是 SPA？尽管 SPA 可能有白屏问题

是的，在 Electron 内部，网页的首选绝对是 SPA（单页应用）。

甚至可以说，SPA 和 Electron 简直是天作之合。在传统 Web 端最让人头疼的“白屏时间（FCP / TTI 慢）”，在 Electron 的本地运行环境下，几乎天然就被消灭了。

我们可以从以下几个维度来拆解为什么 Electron 极度偏爱 SPA，以及如何优雅地彻底干掉那仅存的“本地白屏”问题。

1. 为什么在 Electron 中，SPA 比在 Web 端更爽？

在浏览器里，SPA 被人诟病白屏，是因为用户需要通过网络去下载好几兆的 JS 包。但在 Electron 里，规则全变了：

  • 零网络延迟（Instant Load）： 你的 SPA 资源（HTML、JS、CSS）是直接打包在客户端本地的。Electron 加载网页是通过 file:// 协议或自定义协议读取本地 SSD / HDD。读取一个 5MB 的 JS 包只需要几毫秒，几乎是瞬间完成。
  • 根本不需要 SEO： 桌面应用不需要被百度或 Google 检索，所以 SSR（服务端渲染）最大优势之一（SEO 友好）在 Electron 里完全失去了用武之地。
  • 状态极其稳定： 在桌面应用中，用户期望的操作体验是“顺滑无缝”的。如果采用 MPA（多页应用），每次切页面窗口都会闪烁一下，并且由于页面刷新，内存里的全局变量和 WebSocket 连接会全部中断重连。SPA 的组件级切换能带来最纯正的“原生应用感”。

2. 那么，Electron 里的“白屏”到底是怎么来的？

既然资源都在本地，为什么有时候启动 Electron 还是会看到一瞬间的白屏？

这通常不是 SPA 路由的锅，而是 Chromium 初始化窗口与渲染进程启动的时间差 导致的：

  1. 主进程调用 new BrowserWindow()，操作系统开始绘制窗口。
  2. 此时 Chromium 渲染引擎刚刚启动，本地 HTML 还没来得及加载和解析。
  3. 窗口已经画好了（一片空白），等了几十毫秒后，JS 运行，界面才蹦出来。

3. 终极解法：如何彻底干掉 Electron 的白屏？

在实际工程中，我们有非常成熟的方案来解决这个启动白屏问题。最推荐的做法是 “优雅显现（ready-to-show）”模式。

  3. 用 show: false + ready-to-show 的原生机制，把启动时的白屏时间完全隐藏在幕后。

-webkit-app-region: drag 还是 app-region: drag？文档里两种都有

在 2026 年的今天，如果你是在为 Electron 编写生产环境代码，你应该优先且坚决使用带前缀的 -webkit-app-region: drag。

虽然你可能在某些非官方文档或 CSS 提示中看到了不带前缀的 app-region，但从底层的标准现状来看，它们有着本质的区别。

1. 为什么必须用 -webkit-app-region？

核心原因在于：app-region 并不是一个正式通过的、标准的 W3C CSS 属性。

  • 它是 Chromium 的专属私有特性：这个属性最初是由 Chromium 团队（为了 Chrome Apps 和后来的 Electron、PWA）私有实现的，用来告诉底层的窗口管理器“网页的这一块区域可以用来拖拽窗口”。
  • 前缀保护机制：因为这不是标准属性，Chromium 引擎在实现它时，将其严格限制在了 -webkit- 前缀下。
  • 不带前缀的后果：如果你在 CSS 中写 app-region: drag，Chromium 渲染引擎（Blink）在解析 CSS 时，会直接将其作为非法 / 无法识别的属性过滤掉，导致你的自定义标题栏完全无法拖拽。

2. 编写无框窗口（Frameless Window）拖拽的标准写法

在 Electron 中，实现一个完美的、跨平台无缝拖拽的标题栏，标准的 CSS 写法如下：

避坑指南：为什么我的拖拽区域不能点击了？

当你给一个元素设置了 -webkit-app-region: drag 后，它在底层的行为就不再是一个普通的 HTML 元素，而更像是一个操作系统的原生窗口“把手”。

这会带来两个经典的副作用，你在开发时必须注意：

  1. 鼠标事件被拦截：一旦设为 drag，该元素上所有的鼠标事件（如 click、mouseenter、mouseleave、contextmenu）在渲染进程中将全部失效。因为操作系统会优先把鼠标按下事件当作“拖拽窗口”指令消费掉。
  2. 必须显式排除控制按钮：如果你在标题栏里放了“最小化、最大化、关闭”按钮，必须给这些按钮单独加上 -webkit-app-region: no-drag。否则，用户点击这些按钮时，不仅无法触发点击事件，反而会变成“拖拽窗口”，体验非常糟糕。

总结

在 Electron 开发中，不要猜测和尝试标准 CSS 的写法，请 100% 认准 -webkit-app-region: drag。

// src/main/windows/aboutWindow.ts
import { BrowserWindow } from 'electron';
import * as path from 'path';
// 1. 在模块作用域内持有一个私有引用，用来跟踪当前窗口是否存在
let aboutWindow: BrowserWindow | null = null;

export function openAboutWindow() {
  // 2. 防重点击机制：如果窗口已经存在，直接让它聚焦（提到最前），而不是 new 一个新的
  if (aboutWindow) {
    if (aboutWindow.isMinimized()) aboutWindow.restore();
    aboutWindow.focus();
    return;
  }
  // 3. 窗口不存在，new 一个新的，并进行个性化初始化配置
  aboutWindow = new BrowserWindow({
    width: 380,
    height: 480,
    resizable: false, // 关于窗口通常不需要缩放
    minimizable: false, // 禁用最小化按钮
    maximizable: false, // 禁用最大化按钮
    fullscreenable: false, // 禁用全屏
    title: '关于应用',
    show: false, // 先隐藏，配合 ready-to-show 优雅显现
    webPreferences: {
      preload: path.join(__dirname, '../preload/index.js'),
      sandbox: true
    }
  });
  // 4. 去除原生的顶部菜单栏（仅针对 Windows / Linux，macOS 的菜单在屏幕最上方）
  aboutWindow.removeMenu();
  // 5. 加载特定的本地 Web 页面
  if (process.env.NODE_ENV === 'development') {
    aboutWindow.loadURL('http://localhost:3000/#/about');
  } else {
    aboutWindow.loadFile(
      path.join(__dirname, '../renderer/index.html'),
      { hash: 'about' } // 优雅加载 SPA 的特定子页面
    );
  }
  // 6. 渲染就绪后再显示，防止短暂白屏
  aboutWindow.once('ready-to-show', () => {
    aboutWindow?.show();
  });
  // 7. 极其重要：当窗口关闭时，必须把引用设为 null，否则下次就无法再次打开了！
  aboutWindow.on('closed', () => {
    aboutWindow = null;
  });
}

总结

在 Electron 这种多窗口架构中，new 一个新窗口本身非常廉价，但你需要注意两点：

  1. 防止“多开”： 必须使用局部变量或单例管理器（如 aboutWindow 是否为 null）来守门，如果窗口已存在，应该直接 focus() 把它叫醒。

在 macOS 上，新应用打开都要请求很多权限，比如定位、本地网络、麦克风等；要怎么实现？Windows 上似乎没有权限申请

macOS 和 Windows 在安全和隐私控制哲学上有着极大的差异：

  • macOS 是“严格的沙箱与运行时授权制”：即使你的 App 已经安装成功，任何涉及敏感隐私的 API（定位、麦克风、摄像头、本地网络）在第一次调用时，都必须由操作系统弹窗、经用户手动同意后才能使用。
  • Windows 是“默认信任制 / 安装时授权制”：Windows 传统的 Win32 应用（这也是 Electron 打包出来的默认格式）在运行时拥有极高的特权。一旦用户运行了你的 .exe，它默认就可以直接读写硬盘、调用麦克风和摄像头、扫描本地网络，不需要在运行时弹出系统的隐私确认框。

2. Windows 上真的不需要申请权限吗？

在传统的桌面版 Electron 开发中，答案是：真的不需要。

这是因为：

  1. 传统的 Win32 应用特权极高：Electron 默认打包出来的 .exe 安装包属于传统的桌面应用。在 Windows 的安全模型里，这类应用一旦运行，就默认拥有访问摄像头、麦克风和本地网络的全部权限。
  2. 唯一可能弹窗的场景，防火墙（本地网络）： 当你的 Electron 应用试图在本地建立一个 Socket 服务器（比如启动了一个本地的 Express / Koa 服务，或者进行 P2P 传输），Windows 系统的 Defender 防火墙会弹出一个经典的提示：“Windows 安全中心已阻止此应用的部分功能（允许其在专用 / 公用网络通信）”。
  • 如何解决？ 这不需要你在 Electron 代码里写任何东西。在安装阶段，你可以通过打包工具（如 NSIS 制作的安装包）向系统注册防火墙例外，这样用户在安装时就已经允许了网络通信，运行期间就不会再弹窗了。

  • 开发 Windows 版：保持松弛。你完全可以假定所有硬件权限都是默认开启的。你只需要处理好在极少数由于杀毒软件拦截或防火墙策略导致的调用失败时的 Try-Catch 容错即可。

在 macOS 上申请权限之后，往往需要重启应用；这个重启的过程，是全程系统接管的吗？

在 macOS 上，当用户在“系统设置 ──► 隐私与安全性”中手动为你的应用开启（或关闭）某项敏感权限（比如摄像头、麦克风、辅助功能或完全磁盘访问权限）时，这个重启过程确实是由操作系统底层的安全机制（TCC，即 Transparency, Consent, and Control）深度接管的，但它并不是一个温和的、用户习以为常的“优雅重启”，而是一次无情的“强制终止”。

这里面的运行逻辑和对应用开发的影响非常特殊，我们可以从以下三个层面来剖析：

1. 操作系统的底层接管逻辑：“先斩后奏”

一旦用户在系统设置里勾选或取消了你 App 的某项权限，macOS 会瞬间执行以下操作：

  1. 强行终止进程（SIGKILL / SIGTERM）： macOS 底层的 taskgated 或 tccd 守护进程检测到权限变更，会直接向你的应用发送强制退出信号。这时候，你的 Electron 进程会被操作系统强行杀掉。
  2. 系统弹窗询问（由系统接管）： 此时，macOS 会弹出一个系统级的提示框，通常写着：

    “‘XXX’在重新启动前将无法访问[某权限]。您要现在退出吗？”

  • 如果用户点击 “现在退出”，系统会直接杀掉你的 App 进程（此时应用就处于关闭状态，需要用户手动重新双击打开）。
  • 在某些最新的 macOS 版本或特定权限（如辅助功能 / 截屏录制）下，系统可能只提供一个“退出并重新打开”的单按钮选项。

无论如何，macOS 绝对不会在后台帮你保存应用的状态、不负责帮你重新拉起进程，它只负责最核心的一件事：确保旧的、没有新权限凭证的进程被彻底杀死。

2. 这种“强行接管”会带来哪些工程痛点？

因为系统接管了杀进程的步骤，但不负责帮你恢复状态，这会导致以下几个非常糟糕的用户体验：

  • 数据丢失：由于是被系统强制杀死，Electron 主进程的 app.on('will-quit') 或渲染进程的 window.onbeforeunload 事件可能根本来不及完全执行，导致用户未保存的数据直接丢失。
  • 启动中断：应用突然消失，用户需要自己去应用程序里重新找图标双击，这对于小白用户来说非常让人困惑。

一些应用支持配置网络代理，Electron 怎么实现？

在 Electron 中，实现网络代理配置的核心不在前端，而是在主进程（Main Process）中。Electron 通过 session 模块提供了非常完善的代理控制接口。

对于大多数 Electron 应用来说，你需要处理两种情况：配置代理地址和处理代理认证（账号密码）。

1. 配置代理地址（ses.setProxy）

import { session } from 'electron';
/**
 * 设置代理
 * @param proxyConfig 代理规则对象
 */
async function setProxy(config: { httpProxy: string; bypass: string }) {
  await session.defaultSession.setProxy({
    // 格式："http=127.0.0.1:7890;https=127.0.0.1:7890"
    proxyRules: `http=${config.httpProxy};https=${config.httpProxy}`,
    // 排除列表，不走代理的地址（如本地服务）
    proxyBypassRules: config.bypass || '<local>'
  });
}

关键点

  • 格式要求：proxyRules 必须遵循 Chrome 的代理规则字符串格式。如果多个协议共用一个代理，确保按照 http=...;https=... 的格式书写。
  • 即时生效：调用 setProxy 后，应用内所有的 fetch、axios 请求以及 BrowserWindow 加载的网页请求都会立即切换到新代理，无需重启应用。

模式 B：手动配置 UI（User Config）

如果你的应用像 VS Code 或其他专业工具一样，允许用户在“设置”面板里手动输入代理地址：

  1. 用户保存：前端（渲染进程）通过 ipcRenderer 将代理地址传给主进程。
  2. 主进程持久化：主进程将配置写入 userData 目录下的 JSON 文件（不要放在 localStorage，那是前端的）。
  3. 应用启动时加载：在 app.whenReady() 之后，立即读取该 JSON，调用 session.defaultSession.setProxy()，确保应用启动那一刻代理就生效。

避坑指南

  1. 分 session 代理：如果你在应用里使用了 partition 创建了多个独立的 session（比如为了隔离不同用户的 Cookie），记得给每个 session 分别设置代理，session.defaultSession 只控制默认的那一个。
  2. HTTPS 代理与证书：如果代理服务器使用自签名证书，或者是在做中间人攻击（解密 HTTPS），你可能还需要配置 session.defaultSession.setCertificateVerifyProc 来忽略 SSL 证书错误，否则代理连接会失败。
  3. Bypass Rules 的重要性：永远不要忘记设置 proxyBypassRules。如果不设置，你访问 localhost或公司内部局域网 IP 时，也会强制走代理，这会导致访问失败或极慢。记得加上<local> 和你内部 API 的域名或 IP 段。

在网页中，只能使用 file: 协议请求 ASAR 归档文件中的内容吗？

不是的，在 Electron 中，这其实是一条非常经典且必须要避开的“反模式（Anti-pattern）”。

在渲染进程（网页）中，如果你的应用开启了现代安全策略（如严格的沙箱模式 sandbox: true 或启用了 CSP），直接使用 file:// 协议去请求 ASAR 内部的资源通常会失败，操作系统和 Chromium 的安全沙箱会出于“防范本地文件系统跨域攻击”的机制将其死死拦截。

在现代 Electron 工程中，我们有且仅有一种被官方强烈推荐的黄金方案：自定义特权协议（Custom Protocol）。

官方极力推荐的终极解法：自定义协议（Custom Protocol）

不要让前端直接面对冷冰冰的 file:// 路径。我们可以在主进程中注册一个专属的、安全的特权协议（比如 app:// 或 static://），让它在幕后帮前端去解压和读取 ASAR 文件。

为什么不建议用 file://？（痛点剖析）

如果你强行使用 file:// 协议，你会在开发和生产环境中踩到这三大深坑：

  1. 绝对路径暴露与绝对路径灾难： SPA（特别是像你常用的 Vite 构建的单页应用）默认的资源路径都是绝对根路径（如 /assets/index.js）。如果用 file:// 协议，它会直接去用户电脑的根目录 C:\assets\index.js 找文件，直接导致首屏白屏、资源全部 404。
  2. CSP（内容安全策略）冲突： 为了防范恶意 XSS 攻击，现代高安全性 Electron 应用都会配置 CSP 头部。如果允许 file:// 协议，意味着恶意脚本一旦注入，就能肆无忌惮地读取用户整块硬盘的任何文件。而限制为 app://，恶意脚本就只能读取你 ASAR 包内公开的那几个静态文件。
  3. Web Worker 和 Service Worker 瘫痪： Chromium 处于对隐私的保护，严禁在 file:// 协议下运行 Web Worker 或 Service Worker。如果你的应用需要进行大量的复杂计算或离线缓存，file:// 会让前端技术栈直接断了双腿。

总结

网页中绝对不是只能使用 file: 协议。恰恰相反，在现代 Electron 架构中，使用 protocol.handle 注册一个自定义特权协议（如 app://）才是读取 ASAR 归档文件最标准、最安全、也最不容易让 SPA 路由和路径报错的工业级解法。

为什么文档中使用 file: 协议？

因为对于文档编写者而言，file: 协议是解释 Electron 基本原理时成本最低、最直观的“新手村”方案。

官方文档和很多入门教程之所以在示例中频繁使用 file: 协议（例如 win.loadFile('index.html')，其底层就是转化为 file: 协议加载），主要是出于以下三个原因：

1. 降低新手的认知和上手门槛

如果文档一上来就要求你配置 protocol.registerSchemesAsPrivileged，然后用 protocol.handle 拦截请求，再用 net.fetch 处理文件映射…… 这一整套主进程的底层概念会让刚接触 Electron 的开发者直接“劝退”。

相比之下，file: 协议符合所有网页开发者对“本地双击打开 HTML”的直观认知，能让新手在 3 秒钟内看到第一个窗口跑起来。

2. 在最基础的“纯静态”场景下它确实能跑通

如果你的桌面应用极其简单：

  • 所有的静态资源引用都是相对路径（如 ./style.css，而不是 /style.css）。
  • 没有使用 React / Vue 等现代前端框架的客户端路由（SPA Routing）。
  • 没有配置严格的 CSP（内容安全策略）。

在符合这些极简条件时，直接用 file: 协议加载 ASAR 内部的网页，确实不会报错。文档为了保持示例代码的绝对精简，通常会选择这种没有任何额外配置的“裸跑”方案。

3. 历史包袱与文档更新的滞后性

这是一个非常现实的工程问题。

在 Electron 的早期版本（几年前）中，注册自定义协议（如 protocol.registerFileProtocol 等旧 API）非常繁琐，且在不同版本间频繁发生重构和废弃。由于历史遗留，大量的官方起步教程、Demo（如 electron-quick-start）和第三方博客，至今依然沿用着几年前最古老的 file: 接入方式，并被一代代开发者复制粘贴至今。

官方文档隐藏的“免责声明”

其实，Electron 官方并不是没有警告过。如果你仔细翻阅官方文档中关于 “安全（Security）” 的核心章节，你会发现官方有一条明确的警告：

“不要在生产环境中直接对不信任的内容或复杂的应用放开 file: 协议访问。”

在现代（2026 年）的工业级 Electron 开发中，业界已经形成了一套不成文的默契：

  • 文档 / 教程里的 file:：仅用于快速验证想法、写 Hello World。
  • 实际生产项目：一律使用自定义协议（Custom Protocol）或者本地启动的轻量级安全 Web 服务来托管本地 SPA 资源。

ASAR 完整性检查到底要不要开？

简而言之：只要你的应用要发布给普通用户（生产环境），就一定要开。

这不仅是官方目前强烈推荐的安全最佳实践，更是防御客户端篡改、保护代码资产的第一道核心防线。

1. 为什么要开？不开有什么后果？

Electron 应用打包后，你的核心业务代码（HTML / JS / CSS 以及 node_modules）都会被打包进 app.asar 这个归档文件中。

不开 ASAR 完整性检查（默认状态）

app.asar 本质上只是一个没有任何加密和校验的未压缩的 tar 包。

  • 黑客 / 小白用户可以轻松篡改： 任何人都可以在安装目录下，用一行命令 npx asar extract app.asar ./src 把它解开，修改你的 JS 逻辑（比如绕过你的授权激活检测、注入恶意广告），然后执行 npx asar pack 重新打包。
  • 杀毒软件和系统信任度： 如果你的 ASAR 很容易被注入恶意脚本，那你的应用极易成为本地木马病毒借壳运行的载体，导致应用被杀毒软件误杀。

安全闭环

开启 ASAR 完整性检查（ASAR Integrity）后，打包工具会在编译时计算出 app.asar 的 SHA-256 哈希值，并硬编码进应用的二进制主程序（Windows 的 .exe 资源或 macOS 的二进制文件）。 运行期间，主程序启动时会率先比对两者的哈希值：

一旦发现 app.asar 被拆包动过，主程序会立刻强制闪退（Terminate），不给篡改代码任何运行的机会。

在现代 Electron 开发中，开启完整性检查非常简单，通常配合 Electron Fuses（熔丝机制） 一起启用。

如果你使用的是社区主流的 electron-builder 或 electron-vite，只需要在配置文件中加入几行配置即可。

3. 开了它就高枕无忧了吗？（技术局限性）

作为一名开发者，需要理性认识到：ASAR 完整性检查防御的是“代码篡改（Tampering）”，而不是“代码逆向（Reverse Engineering）”。

即便开了完整性检查，你的核心 JS 源码依然是明文躺在 ASAR 包里的。如果别人只是想看你的业务逻辑或提取你的 API Key，直接解包出来阅读即可。

工业级安全闭环

如果你要做到真正的“防篡改 + 防逆向”两手抓，业界标准的做法是：

  1. ASAR 完整性检查（ASAR Integrity）：守住入口，防止代码被篡改并重新运行。
  2. V8 字节码编译（V8 Bytecode）：通过工具（如 electron-vite 提供的字节码保护）将你的 JS / TS 编译为 V8 引擎能直接识别的二进制字节码（.jsc），彻底消灭可读的明文 JS 源码。

Electron 中的 ES 模块支持顶层 await 了吗？

在 2026 年的今天，答案是：支持，但这里面有一个非常重要且容易踩坑的「分层支持」情况。

Electron 内部由 Node.js（主进程） 和 Chromium（渲染进程） 两个完全不同的 Runtime 驱动。虽然它们目前都已经完全具备了对 ES 模块（ESM）及顶层 await（Top-level await） 的原生支持，但在实际工程落地中，两者的使用限制大不相同。

1. 渲染进程（网页端）：100% 完美支持

在你的前端代码或者 Preload 脚本中，顶层 await 已经非常成熟。

只要你的现代前端构建工具（如 Vite）输出的是标准的 ESM 格式，或者你在 HTML 中使用了 <script type="module">，你就可以肆无忌惮地使用它。这对于在页面加载前先去异步请求一些配置数据极其方便：

import { createApp } from 'vue';

2. 主进程（Node.js 端）：支持，但需要迈过「工程坎」

在 Electron 的主进程（Main Process）中，底层使用的是 Node.js。Node.js 在很早的版本就已经原生支持了 ESM 和顶层 await，但是你在 Electron 主进程中开启它，会面临两个棘手的问题：

问题 A：package.json 的 type: "module" 冲突

要在 Node.js 中使用顶层 await，文件必须被识别为 ES 模块。通常的做法是在 package.json 中配置 "type": "module"。

  • 坑点： 一旦你开启了这个全局配置，你项目中所有未经编译的第三方 Node 模块如果只支持 CommonJS，都会直接报错爆掉。

最佳工程解法：让 Vite / Rspack 在编译时抹平差异

在实际企业级开发中，我们几乎从来不在主进程中直接裸跑原生的 ESM 文件。

像你平时使用的现代 Electron 构建工具（如 electron-vite），在打包主进程时，通常会通过底层的 Rollup / Esbuild 将主进程的 TS 代码转换为符合 CommonJS 规范的单文件，或者通过特殊的打包配置输出符合特性的 ESM 包。

如果你使用的是 Vite + 现代配置，你可以在主进程打包中直接书写顶层 await，Vite 会帮你安全地编译好：

避坑指南：主进程顶层 await 的「生命周期死锁」

虽然技术上支持，但在主进程的顶层（Top-level）使用 await 时，必须死死记住一个心智模型：

主进程的顶层 await 会直接阻塞整个 Electron 应用程序的生命周期！

如果你在主进程文件的最顶层写了一个需要耗时 5 秒钟的网络 fetch 请求，这意味着在网络请求完成之前，app.whenReady() 永远不会被触发，你的 App 看起来就像是死锁闪退或卡死在后台，操作系统甚至会认为你的程序已经无响应。

最佳安全准则

  • 渲染进程：随便用，通常只用来处理路由或轻量配置阻塞。
  • 主进程：顶层 await 只用于耗时极短的本地操作（如读取一个几十 KB 的本地 JSON、初始化本地轻量数据库连接）。任何涉及网络请求、耗时文件扫描的异步逻辑，依然必须乖乖放进 app.whenReady().then(async () => { ... }) 的内部生命周期中去执行。

为什么 Preload Script 必须使用 .mjs？而其他的不用？

在 Electron 现代安全架构的演进中，Preload 脚本的格式限制实际上并不是 Electron 官方“强制”要求的，而是现代构建工具（特别是 Vite）和 Chromium 沙箱策略共同夹击下的工程选择。

你之所以会发现 Preload 脚本必须叫 .mjs（或者在构建后输出为 ESM 格式），而主进程和渲染进程可以相对自由，是因为 Preload 脚本正好处在一个极其特殊的“夹缝环境”中。

1. 根源：Preload 脚本被夹在两个世界中间

我们要先看 Electron 目前最核心的安全基石：contextIsolation: true（上下文隔离）。

在这个模式下，Preload 脚本的运行环境非常奇特：

  • 它能访问 Node.js API： 它有特权去调用 ipcRenderer，甚至在配置允许时调用 fs、path 等。
  • 它运行在 Chromium 的独立沙箱里： 它与网页（渲染进程）的 JavaScript 是完全隔离的。

在这个特殊的沙箱环境中，Chromium 引擎对脚本的模块化加载有着极其严苛的限制。

2. 为什么 Preload 不能像网页那样随意？

为什么不能用 CommonJS（.js/require）？

随着 Electron 和 Chromium 对沙箱（Sandbox）的收紧，现代 Preload 脚本在开启沙箱时，是严禁在运行时使用底层的 require() 语法的。因为原生的 require 是同步读取文件系统的，这会直接破坏 Chromium 的安全沙箱规则。因此，传统的 CommonJS 格式在现代 Preload 中直接被判了死刑。

为什么不能像网页（渲染进程）那样交给打包工具二合一？

在网页端（渲染进程），像 Vite 这样的工具可以把你的 React / Vue 代码、路由、状态管理全部打包融合成一个巨大的 index.js，甚至可以通过本地自定义协议（如 app://）以 ESM 模块的方式用 <script type="module"> 异步加载。

但 Preload 脚本不行。Preload 必须在网页的第一行任何代码执行之前，由 Electron 主进程强行注入到 Chromium 的置顶上下文中。它必须是一个独立的、符合规范的、可被 Chromium 引擎直接解析的单个文件。

3. 为什么是 .mjs（ESM）？

为了让 Preload 脚本既能安全地导入模块（比如导入自定义的类型、工具函数），又能通过 Chromium 的安全审查，它必须输出为标准的 ES 模块（ECMAScript Modules）。

在 Node.js 和很多现代构建工具的默认规则中：

  • .js：默认被当成古老的 CommonJS（CJS）处理。
  • .mjs：强制标记为现代的 ES 模块（ESM）。

当 Vite 看到你的 Preload 脚本是 .mjs，或者你配置了将其输出为 .mjs 时，Vite 会采用专用的打包策略：

  1. 树摇与合并：把你在 Preload 里引用的第三方库或辅助函数全部打包合并成一个孤零零的纯纯的 ES 模块文件。
  2. 使用标准 import / export：这样生成的代码里只有标准的 import，没有 require。当 Electron 把这个 .mjs 喂给 Chromium 时，Chromium 的沙箱引擎能完美识别并安全执行。

为什么“主进程”和“渲染进程”不用这么麻烦？

我们可以对比一下这三者的编译待遇：

  • 渲染进程（网页）：Vite 专门为它服务。Vite 本身就是基于 ESM 的，网页端怎么写，Vite 都能通过开发服务器（Dev Server）或生成标准的 index.html 完美托管，不需要刻意改成 .mjs。
  • 主进程（Node.js）：主进程运行在纯粹的 Node.js 环境中，不碰 Chromium 沙箱。Vite 打包主进程时，通常会通过底层配置（如打包成单个 .js 文件，内部自动处理好 require），Node.js 吞下 CommonJS 毫无压力。
  • Preload 脚本：姥姥不疼老爷不爱。它既要满足 Vite 的 ESM 编译链，又要满足 Chromium 苛刻的沙箱注入格式。因此，显式地使用 .mjs 扩展名，是目前工程上通知构建工具和运行时“这货是一个纯净的、无 require 的原生 ES 模块”最稳妥、最不容易报错的手段。

总结

在现代 Electron 项目中，把 Preload 脚本配置为 .mjs 输出，本质上是为了顺应 Chromium 沙箱去 CommonJS 化的趋势，同时让 Vite 能够正确地对其进行 ESM 格式的单文件打包。

文档注意事项中的「您必须在 app 的 ready 事件之前使用 await」是什么意思？

这句话在技术上其实有一点小小的语义歧义，但在 Electron 的语境下，它是在向你强调一个非常关键的顺序问题：

它的真正意思是：“如果你有某些核心的全局配置、特权协议注册，你必须在 app.whenReady() 被触发之前就完成它们的执行（包括等待它们的异步结果），绝对不能等到 ready 事件之后才去做”。

这里的 await 通常指代的是阻塞后续生命周期，确保配置率先生效的动作。我们来看看为什么官方要如此严厉地强调这个顺序。

1. 核心原因：有些 API 属于“投胎型”API

Electron 的生命周期非常严格。有一些基础的底层配置 API，被戏称为“投胎型 API”；它们必须在 Electron 刚一睁眼、还没有初始化完 Chromium 内核和网络栈之前就完成注册。一旦错过了 ready 那个时间节点，它们就彻底失效了。

最典型的代表就是我们前面聊到的：protocol.registerSchemesAsPrivileged（注册自定义特权协议）。

2. 还有哪些东西必须在 ready 之前处理？

除了自定义特权协议，以下这些敏感操作也强烈建议（或必须）在 app.whenReady() 之前准备就绪：

  1. 单实例锁检测 app.requestSingleInstanceLock()： 你必须在最顶层去判断当前是不是已经有一个相同的 App 正在运行了。如果是，直接退出，根本不需要等到 ready 去浪费内存初始化应用。
  2. 崩溃报告初始化 crashReporter.start()： 如果在 ready 之前程序就因为硬件兼容性崩了，只有在最顶层初始化的崩溃报告器才能抓到这个早期崩溃。
  3. 部分硬编码的命令行开关 app.commandLine.appendSwitch： 比如禁用硬件加速、配置特殊的 Chrome 实验性功能，必须在进程刚启动时塞进命令行参数里。

总结

文档中说这句话的目的，是为了防止你把初始化代码一股脑全塞进 app.whenReady().then(...) 的大括号里。

理解这个概念的心智模型很简单：app.whenReady() 标志着“地基已经全部打好，水泥已经彻底干了”。所有用来改变地基属性的图纸和材料（如自定义协议、命令行开关），你必须在水泥倒下去（ready 触发）之前，用代码（或顶层 await）把它们递给操作系统。
