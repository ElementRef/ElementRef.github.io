Next.js 之 DataFetching

Aug 18, 2024
阅读时长：7 分钟

在不需要实时更新数据的场景下，使用 ServerComponents 在服务端完成数据获取比较好：

  • 请求来的数据会被缓存（在 Next 15 中，缓存行为 有变 ），并在适当的时候验证缓存；
  • 避免敏感信息（token、APIKeys）被暴露在客户端
  • 减少客户端数据请求的次数（数据请求瀑布）；
  • 减少数据获取延迟。

相应的，他也有一些缺点：

  • 需要实时展示的数据（一般对 SEO 也不重要），在客户端请求比较好；
  • 需要额外跑 Node 服务。

在 AppRouter 模式，不再有 getServerSideProps、getStaticProps 和 getInitialProps 了；现在，可以直接在 ServerComponents 中发起异步请求。

对应的 getStaticPaths 也被新的 generateStaticParams 取代。

// app/posts/[id]/page.js
import PostLayout from '@/components/post-layout';
export async function generateStaticParams() {
  return [{ id: '1' }, { id: '2' }];
}
async function getPost(params) {
  const res = await fetch(`https://.../posts/${params.id}`);
  const post = await res.json();
  return post;
}
// 和之前一样，generateStaticParams 的返回值会被注入到 props 中
export default async function Post({ params }) {
  const post = await getPost(params);
  return <PostLayout post={post} />;
}

getStaticPaths 中的 fallback 属性被现在的 RouterConfig 取代。

getStaticPaths 中的 fallback 属性有三个取值：

  • true，请求到达服务端时，动态生成新页面，并先返回 FallbackPage；
  • false，返回 404 页面，等同于 export const dynamicParams = false；
  • blocking，请求到达服务端时，动态生成新页面，不再支持。

true 和 blocking 选项被统一为 export const dynamicParams = true；同时 getStaticPaths 中的 revalidate 属性被合并到 fetch 中：

export const dynamicParams = true;
async function getPosts() {
  const res = await fetch(`https://.../posts`, { next: { revalidate: 60 } });
  const data = await res.json();
  return data.posts;
}
export default async function PostList() {
  const posts = await getPosts();
  return posts.map(post => <div>{post.name}</div>);
}

// app/artist/[username]/page.tsx
import Albums from './albums';
async function getArtist(username: string) {
  const res = await fetch(`https://api.example.com/artist/${username}`);
  return res.json();
}
async function getAlbums(username: string) {
  const res = await fetch(`https://api.example.com/artist/${username}/albums`);
  return res.json();
}
export default async function Page({
  params: { username }
}: {
  params: { username: string };
}) {
  const artistData = getArtist(username);
  const albumsData = getAlbums(username);
  const [artist, albums] = await Promise.all([artistData, albumsData]);
  return (
    <>
      <h1>{artist.name}</h1>
      <Albums list={albums} />
    </>
  );
}

数据验证

数据验证可以通过两个方式：

  • 基于时间的验证，在缓存被创建的一段时间之后，如果再次发生请求，则先对缓存进行验证；
  • 按需验证，基于特定事件（例如表单提交），手动触发数据验证。

对于静态渲染的路由，如果有多个请求，并且每个请求配置了不同的 revalidate；那么对于这些请求来说，实际的 revalidate 为最小的 revalidate。

对于动态渲染的路由，每个请求的 revalidate 是独立的。

如果数据时效性不强，可以配置尽可能长的 revalidate 以节省服务器资源；如果数据时效性比较强，可以改为客户端渲染。

1
2
3

fetch('https://...', { next: { revalidate: 3600 } });
// 或者在 layout.js | page.js 中导出 revalidate
export const revalidate = 3600;

可以使用 revalidatePath 和 revalidateTag 来完成按需验证；这两个 API 适用于 RouteHandlers 和 ServerActions 中（这不就是 ReactQuery 做的事情吗？）。

  // ...

}

如果数据验证过程中出错，那么缓存不会被更新；在下一次请求中，会重新进行数据验证。

ServerActions

ServerActions 是一个在服务器上执行的异步函数，可以在 ServerComponents 和 ClientComponents 中调用；用来处理表单提交和数据修改逻辑；使用 ‘use server’ 声明。

('use client');

  updateItemAction
}: {
  updateItemAction: (formData: FormData) => void;
}) {

  • ServerActions 可以直接绑定在 form 表单（React 对 form 做了扩展）的 action 属性上，并且在表单提交时，不会刷新页面；
  • ServerActions 可以绑定在事件处理器、useEffect、任何表单元素（例如 button）上；
  • ServerActions 从 layout.tsx / page.tsx 中继承路由配置；
  • ServerActions 集成了 Next.js 的缓存和数据验证机制；
  • ServerActions 的入/出参必须可以被 序列化 ；
  • ServerActions 仅支持 POST 方法。

'use client';
export function Entry() {
  const handleKeyDown = (e: React.KeyboardEvent<HTMLTextAreaElement>) => {
    if (
      (e.ctrlKey || e.metaKey) &&
      (e.key === 'Enter' || e.key === 'NumpadEnter')
    ) {
      e.preventDefault();
      e.currentTarget.form?.requestSubmit();
    }
  };
  return (
    <div>
      <textarea name="entry" rows={20} required onKeyDown={handleKeyDown} />
    </div>
  );
}

// app/api/messages/route.ts
export async function GET() {
  const data = await fetch('https://...', {
    headers: {
      'Content-Type': 'application/json',
      'API-Key': process.env.DATA_API_KEY
    }
  }).then(res => res.json());
  return Response.json({ data });
}

一般 ServerComponents 不需要调用 RouteHandlers ，因为他可以直接在服务端完成数据请求。

ServerActions 与 RouteHandlers 区别

  • 两者都是用来写可以在客户端调用的服务端逻辑的；
  • RouteHandlers 偏向于底层一些，可以用来写 RESTful 风格的服务端代码；
  • ServerActions 更偏向于上层，主要用来创建/修改数据（从仅支持 POST 也能看出来）。
