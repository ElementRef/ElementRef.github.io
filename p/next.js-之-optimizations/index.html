Next.js 之 Optimizations

Aug 19, 2024
阅读时长：7 分钟

<Images/>

ImageComponent 扩展了原生 <img/> 标签：

  • 为每种设备提供适合大小的图片（srcset），提供了 WebP 和 AVIF 格式；
  • 在图片加载时，自动阻止内容偏移，优化 CLP；
  • 图片懒加载，并提供模糊占位；
  • 按需调整图片大小。

import Image from 'next/image';
/**
 * 图片不能被异步导入：import()/require()
 * 因为图片需要在打包时被静态分析
 */
import profilePic from './me.png';
export default function Page() {
  return (
    <Image
      src={profilePic}
      alt="Picture of the author"
      // width={500} 自动生成
      // height={500} 自动生成
      // blurDataURL="data:..." 自动生成
      // placeholder="blur" // 可选属性，模糊占位
    />
  );
}

如果图片来自远程 URL，width、height、blurDataURL、placeholder 属性需要手动设置；width 和 height 是指图片本身的宽和高，只是为了在加载时，能正确推断图片宽高比，不决定图片的具体大小，避免内容偏移。

ImageComponent 支持 priority 属性，用来控制加载优先级，以便于 Next.js 能对 LCP 进行优化：

1
2
3
4
5

import Image from 'next/image';
import profilePic from '../public/me.png';
export default function Page() {
  return <Image src={profilePic} alt="Picture of the author" priority />;
}

Next.js 真的很重视 CLP（累计内容偏移），优化这个指标最好的方式就是写死图片大小：

  • 在 ImageComponent 上静态导入本地图片；
  • 对于外部图片，使用 width、height 属性；
  • 使用 fill 属性，使得图片去填充父级元素（position:relative && display:block）。

import Script from 'next/script';

  children
}: {
  children: React.ReactNode;
}) {
  return (
    <>
      <section>{children}</section>

这样，当访问 dashboard/page.tsx 或嵌套路由 dashboard/settings/page.tsx 时，脚本将被加载；并且在页面导航中，只会加载一次。

next/script 对第三方脚本有四种加载策略：

  • beforeInteractive，先于 Next.js 代码，发生在 Hydrate 之前；
  • afterInteractive，默认值，在 Hydrate 之后加载脚本；
  • lazyOnload，在浏览器空闲的时候加载脚本；
  • worker，在 WebWorker 中加载。

<Script/> 上有三个事件，只能运行在客户端：

  • onLoad，脚本加载完毕；
  • onReady，组件挂载完毕并且脚本加载完毕；
  • onError，脚本加载出错。

// app/layout.tsx
import { Roboto_Mono } from 'next/font/google';
const roboto_mono = Roboto_Mono({
  // 参考 https://developer.mozilla.org/zh-CN/docs/Web/CSS/@font-face/font-display
  display: 'swap',
  style: ['normal', 'italic'],
  subsets: ['latin'],
  weight: '400' // 如果是可变字体，不需要指定字重
  weight: ['400', '700'], // 字重也可以是一个数组
  preload： true // 默认为 true，是否对字体进行预加载
});
export default function RootLayout({
  children
}: {
  children: React.ReactNode;
}) {
  return (
    <html lang="en" className={roboto_mono.className}>
      <body>{children}</body>
    </html>
  );
}

next/font/google 与 next/font/local 的区别：

next/font 方法被调用时，字体预加载逻辑，和调用的位置有关：

  • 仅在某个 page.tsx 中调用，字体仅对当前路由预加载字体；
  • 在某个非顶级 layout.tsx 中调用，为当前 layout 下的所有路由预加载字体；
  • 在顶级 layout 中调用，为所有路由预加载字体。

注意， next/font 不是单例模式，每次调用都会生成一个实例；所以，应该在共享文件中调用并导出实例，在用到的地方引入实例。

      <p className={inter.className}>Hello world using Inter font</p>

      </p>
      <p className={greatVibes.className}>My title in Great Vibes font</p>

Metadata

一个网站有两种元数据：

  • <meta/>，在 layout.tsx 或 page.tsx 中使用 generateMetadata 函数；或者导出一个静态的 metadata 对象；
  • 为网站添加一些静态或者动态渲染的文件，优先级较高：
    • favicon.ico ；
    • manifest.json ；
    • opengraph-image.jpg ；
    • robots.txt ；
    • sitemap.xml 。

新的 Metadata 用来取代 PagesRouter 中的 next/head。

// app/products/[id]/page.tsx
import type { Metadata, ResolvingMetadata } from 'next';
type Props = {
  params: { id: string };
  searchParams: { [key: string]: string | string[] | undefined };
};
export async function generateMetadata(
  { params, searchParams }: Props,
  parent: ResolvingMetadata
): Promise<Metadata> {
  const id = params.id; // 读取 params
  const product = await fetch(`https://.../${id}`).then(res => res.json()); // 获取相关信息
  // 可以基于父级元数据进行扩展
  const previousImages = (await parent).openGraph?.images || [];
  return {
    title: product.title,
    openGraph: {
      images: ['/some-specific-page-image.jpg', ...previousImages]
    }
  };
}
export default function Page({ params, searchParams }: Props) {}
  • StaticMetadata 和 DynamicMetadata 仅支持 ServerComponents ；
  • generateMetadata 和 generateStaticParams 中的 fetch 会有 RequestMemoization 优化；
  • 为了让 <head/> 最先被传至客户端， generateMetadata 会阻塞流式渲染。

// app/page.js
'use client';
import { useState } from 'react';
import dynamic from 'next/dynamic';
// 将懒加载的 ClientComponents 包装一下
const ComponentA = dynamic(() => import('../components/A'));
// 加载的过程中，会展示 loading
const ComponentB = dynamic(() => import('../components/B'), {
  loading: () => <p>Loading...</p>
});
// ssr:false 表示当前组件不会被预渲染，只会在客户端渲染
const ComponentC = dynamic(() => import('../components/C'), { ssr: false });
export default function ClientComponentExample() {
  const [showMore, setShowMore] = useState(false);
  return (
    <div>
      {/* 会立即加载，但是代码被分割在单独的 bundle 里 */}
      <ComponentA />
      {/* 根据 showMore 动态加载 */}
      {showMore && <ComponentB />}
      <button onClick={() => setShowMore(!showMore)}>Toggle</button>
      {/* 只会在客户端渲染 */}
      <ComponentC />
    </div>
  );
}

    const body = JSON.stringify(metric);

    switch (metric.name) {
      case 'FCP': {
        // handle FCP results
      }
      case 'LCP': {
        // handle LCP results
      }

    }
  });
}

静态导出

默认情况下，在执行 next build 的时候，Next.js 会为每个路由生成静态的 HTML 文件；这种方式可以减少客户端运行的代码，提升加载速度。

如果没有使用动态数据的场景，可以通过配置，使得 Next.js 渲染出一个静态网站/SPA：

1
2
3
4
5
6

// next.config.js
const nextConfig = {
  output: 'export' // 在老版本中，使用 next export 替换
  // distDir: 'dist', // 可选，将输出目录改为 dist
};
module.exports = nextConfig;

这种模式下：

  • ServerComponents 内部的异步请求，会在打包阶段进行；组件运行结果会被保存为静态 HTML，在页面初始渲染或路由跳转时使用；
  • next/image 需要借助自定义 loader 完成图片优化；loader 内部定义了如何从远程获取图片资源；
  • RouteHandlers 会在打包时运行，并且仅支持 GET 方法；
  • ClientComponents 在打包阶段被预渲染为 HTML；需要在客户端渲染完成之后才能访问浏览器 API。

认证

一个完整的前后端系统一定包含这三个东西：

  • 认证，验证用户是不是他自己，需要一些凭证来证明，比如说：用户名和密码；
  • 会话，用来在请求中，追踪用户认证状态；
  • 权限，规定了用户所能访问到的数据范围。
