文档上关于 Dark Mode 的例子，只有 macOS 事例，Windows 一样吗？

简单来说：核心的 API 设计是一样的，但系统底层表现有细节差异，需要你在 Windows 上额外做一点适配。

在 Electron 中，控制暗黑模式的主力 API 都是 nativeTheme 模块（主要使用 nativeTheme.themeSource 和 nativeTheme.shouldUseDarkColors）。

但在实际落地到 Windows 时，你需要注意以下两个关键的不同点和最佳实践。

1. 核心 API 表现一致

在 macOS 和 Windows 上，你都可以用同样的方式去强制或同步系统主题：

import { app, BrowserWindow, nativeTheme } from 'electron';
// 1. 手动强制设为暗黑模式
nativeTheme.themeSource = 'dark'; // 'light' | 'dark' | 'system'
// 2. 监听系统主题变化（Windows 同样支持此事件）
nativeTheme.on('updated', () => {
  const isDark = nativeTheme.shouldUseDarkColors;
  console.log(`当前是否为暗黑模式：${isDark}`);
  // 此时可以通知渲染进程更新 CSS
});

在渲染进程（Renderer）中，CSS 的媒体查询 @media (prefers-color-scheme: dark) 在 Windows 下同样会随着 themeSource 的改变而自动响应，这和 macOS 完全一致。

我们先拆解设备唯一 ID 在 Mac 上的坑，然后再看如何优雅地自建激活服务。

一、 node-machine-id 在 Apple 设备下可靠吗？

结论：不完全可靠，在 macOS 下有失效和变化的风险。

node-machine-id 获取 Mac 唯一标识的原理，是读取系统底层的 IOPlatformUUID（即硬件 UUID）。

在 macOS 上，它相当于在终端执行：ioreg -rd1 -c IOPlatformExpertDevice | grep "IOPlatformUUID"。

虽然这个值是由苹果主板（或者是 Apple Silicon 的 Secure Enclave 安全隔离区）硬件决定的，看起来很稳定，但在以下场景中会遇到问题：

1. 权限与沙盒限制（最致命的坑）

如果你计划未来将 macOS 应用提交到 Mac App Store（必须开启 Sandbox 沙盒模式），沙盒政策会直接拦截对 IOPlatformExpertDevice 的访问。此时 node-machine-id 会报错或返回一串无意义的默认值，导致激活校验直接失败。

2. Apple Silicon 虚拟机隔离

随着 M1 / M2 / M3 / M4 系列芯片普及，越来越多开发者在 Mac 上通过 Parallels Desktop 或 UTM 等虚拟机运行 macOS。在虚拟机环境中，硬件 UUID 可能会在虚拟机重启、备份恢复或软件更新时发生变化，导致用户无缘无故被“反激活”。

3. 主板维修

如果用户的 Mac 因为故障去 Apple Store 更换了主板，它的 IOPlatformUUID 会彻底改变，用户必须联系你重置激活。

import { safeStorage } from 'electron';
import { randomUUID } from 'crypto';
import Store from 'electron-store'; // 推荐使用此库存储配置

const store = new Store();

function getOrCreateClientId(): string {
  // 1. 尝试从本地配置文件读取已加密的唯一 ID
  const encryptedId = store.get('client_id_encrypted') as string;
  if (encryptedId) {
    try {
      // 解密。safeStorage 在 Mac 上绑定系统 Keychain，在 Windows 上绑定 DPAPI
      // 即使别人拷走配置文件，在另一台电脑上也无法解密，安全性极高
      return safeStorage.decryptString(Buffer.from(encryptedId, 'hex'));
    } catch (e) {
      console.warn('解密失败，可能是在新设备上运行，准备重新生成...');
    }
  }
  // 2. 如果是首次运行（或更换了机器解密失败），生成一个新的 UUIDv4
  const newId = randomUUID();
  const encrypted = safeStorage.encryptString(newId).toString('hex');
  store.set('client_id_encrypted', encrypted);
  return newId;
}

当用户尝试打开 App 的第二个实例时，怎么阻止这种行为？Windows / macOS 都会触发 second-instance 事件吗？

在 Electron 中，阻止多实例（单例模式）的实现机制在 Windows 和 macOS 上有着本质的底层差异。

结论是：Windows 会完美触发 second-instance 事件，但 macOS 的机制截然不同。如果你只写了 second-instance 监听，macOS 版本其实留下了漏洞。

下面我们拆开来看为什么，以及如何写出一套真正两端通用的完美单例代码。

1. 为什么两端表现不一样？（底层机制）

Windows：完全依赖进程唤醒

Windows 操作系统本身允许用户通过双击 .exe 文件无限制地创建新进程。 当用户双击第二个 .exe 时，Electron 的 app.requestSingleInstanceLock() 会去检查系统全局锁：

  1. 它发现已经有一个实例在运行了，于是第二个实例会立刻把它的命令行参数（argv）发送给第一个实例。
  2. 随后，第一个实例收到信号，触发 second-instance 事件。
  3. 第二个实例随即自我销毁 app.quit()。

macOS：系统天然限制，但在特殊情况下失效

macOS 默认的 Dock 栏设计就是单例的。当你点击 Dock 栏的图标时，系统的 LaunchServices 如果发现应用已经打开，只会把原应用聚焦到前台，根本不会去启动第二个进程。

  • 因为压根没启动新进程，所以 macOS 正常情况下绝对不会触发 second-instance 事件！

但是！以下两个场景会打破 macOS 的默认机制：

  • 用户在终端（Terminal）里使用绝对路径强行启动：/Applications/MyApp.app/Contents/MacOS/MyApp。
  • 或者是通过外部网页的 Deep Link 链接（自定义协议）连续唤醒你的 App。

在这两种特殊情况下，macOS 会启动新进程，并且会触发 second-instance 事件。

  // 当前实例（第二实例）直接闪退退出
  app.quit();
} else {
  // 2.【Windows / Mac 兜底方案】当第二个实例被强行启动时，主实例会收到这个事件
  app.on('second-instance', (event, commandLine, workingDirectory) => {
    if (mainWindow) {
      // 如果窗口被最小化了，将其恢复
      if (mainWindow.isMinimized()) mainWindow.restore();
      // 将窗口移到最前方并聚焦
      mainWindow.focus();
    }
  });
  // 3.【常规主流程】当应用初始化完成时创建窗口
  app.whenReady().then(() => {
    createMainWindow();
    // 4.【macOS 专属】当点击 Dock 栏图标时激活应用
    app.on('activate', () => {
      // 针对 Mac 系统的特有习惯：如果没有任何窗口活着
      // 点击 Dock 栏应该重新创建一个窗口
      if (BrowserWindow.getAllWindows().length === 0) {
        createMainWindow();
      } else if (mainWindow) {
        // 如果窗口本来就活着，点击 Dock 栏直接聚焦它
        mainWindow.focus();
      }
    });
  });
}
// 正常的退出处理
app.on('window-all-closed', () => {
  if (process.platform !== 'darwin') {
    app.quit();
  }
});

3. 注意一个小坑：开发环境（Dev）下的 Windows 调试

在 Windows 开发环境下，如果你使用 npm run dev（通常背后运行的是 electron .），当你尝试开第二个实例进行单例测试时，你会发现 second-instance 有时无法被触发，而是直接弹了两个一模一样的窗口。

这是因为在开发环境中，两个实例的宿主进程都是全局的 electron.exe，命令行参数里的脚本路径不同，可能会导致 Electron 误判它们不是同一个应用。

1. 开发环境（process.defaultApp === true）

我们可以用 process.defaultApp 来判断是否是开发环境？

答案是：可以，但在实际的项目工程中，它并不是最稳妥和最常用的方法。

process.defaultApp 的确是 Electron 官方提供的一个属性。它的设计初衷非常单纯：如果当前应用是通过 Electron 默认的预编译外壳（即 electron 或 electron .）启动的，该值为 true；如果是通过打包后的自定义可执行文件启动的，该值为 undefined。

这使得它在处理类似“Deep Link 注册路径”这种与“是否使用官方外壳程序启动”强相关的场景时非常完美。

为什么不推荐直接用 process.defaultApp 判断开发环境？

1. 概念不等价：在打包后测试时会失效

当你执行完打包（如使用 electron-builder 或 electron-forge 输出了 dist/win-unpacked/MyApp.exe），你想在本地测试一下打包后的版本是否有 Bug。

  • 此时你运行的是打包后的 .exe，process.defaultApp 会变成 undefined（代表生产环境）。
  • 但对你而言，这依然是开发 / 测试阶段。如果你在这个阶段需要打印详细的 console.log、开启 Chrome DevTools、或者连接本地的测试服务器（localhost:3000），用 process.defaultApp 就会导致应用直接切换到正式环境，无法调试。

2. 它是只读的，无法灵活控制

process.defaultApp 是由 Electron 运行时内部控制的，你无法在启动脚本中手动修改它。我们无法通过配置环境变量（如 NODE_ENV=development）来人为干预它的值。

Deep Links 和 URL Scheme 以及 Universal Link 是一种东西吗？

简单来说：它们不是同一个维度的概念。

用一句话总结它们的关系：Deep Links（深度链接）是一种“设计概念和目标”，而 URL Scheme 和 Universal Link 则是实现这个目标的两种“具体技术手段”。

我们可以把它们放在一个清晰的层级框架里来看：

1. 什么是 Deep Links（深度链接）？

这是一个广义的、抽象的概念。 普通的链接（如 https://example.com）只会带你到网站首页。而 Deep Link 指的是任何能够“直接穿透到应用深处、展示特定页面/内容”的链接。

无论技术上是用自定义协议（如 my-app://book/123），还是用标准网页链接（如 https://example.com/book/123），只要能直达具体内容，它们在产品设计上都叫 Deep Links。

2. 什么是 URL Scheme（自定义协议）？

这是实现 Deep Link 最传统、最古老的技术。上文我们在 Electron 例子中使用的就是它。

  • 格式： my-app://path/to/page
  • 原理： 你的 App 注册了一个独特的协议头（比如 my-app）。当系统收到带有这个协议头的请求时，就会把请求分发给你的 App。
  • 痛点：
    1. 极易冲突： 任何 App 都可以自称是 my-app://。如果在同一台电脑上安装了两个同样注册该 Scheme 的软件，系统会感到困惑（在 Windows 上，后安装的通常会直接覆盖抢走前者的注册表项）。
    2. 糟糕的降级体验： 如果用户没有安装你的 App，直接在浏览器输入 my-app://...，浏览器会生硬地弹出一个报错提示（如“无法打开此网页”或“未找到对应应用”），体验非常差。

3. 什么是 Universal Link（通用链接）？

这是苹果（iOS 9+ / macOS）为了解决 URL Scheme 的痛点，推出的一种更安全、体验更完美的深度链接技术。

  • 格式： https://example.com/path/to/page（它就是普通的、标准的 HTTPS 链接！）
  • 原理：
    1. 你的域名（比如 example.com）下配置了一个由你控制的配置文件（apple-app-site-association）。
    2. 用户在设备上安装了你的 App 后，操作系统会自动去该域名下下载这个文件，并建立“域名 ◄──► App”的本地双向绑定。
    3. 此时，当用户点击这个标准的 https 链接，系统不会打开浏览器，而是直接无缝唤醒你的 App。
  • 优势：
    1. 绝对安全唯一： 因为你必须拥有 example.com 域名所有权才能上传配置文件，其他流氓 App 绝对无法仿冒抢占这个链接。
    2. 完美的无缝降级： 如果用户没有安装 App，点击这个链接就会像普通网页一样在浏览器中打开你的官方网站，完全不会报错。

补充： 在 Android（安卓）生态中，和 Universal Link 对应的一套一模一样的技术，被称为 App Links。

总结：两者的多维度对比

维度URL Scheme（自定义协议）Universal Link（通用链接 / App Links）
链接长相my-app://book/123https://example.com/book/123
安全唯一性低（任何应用都可抢占注册）极高（与网站域名双向强绑定）
未安装应用时的表现浏览器直接报错，体验中断优雅降级，直接在浏览器打开该网页
配置复杂度极低（在应用内写几行代码或改改注册表 / plist 即可）中等（除了客户端，还需在 HTTPS 服务器上部署配置文件）
适用平台macOS / Windows / Android / iOS 全平台仅适用于移动端（iOS / Android）及部分 macOS App

在 Electron（桌面端）的实际场景中： 因为 Windows 平台目前并没有一套和苹果 Universal Link 完全对应且流行的桌面端机制，所以我们在 Electron 中，绝大多数时候依然使用 URL Scheme 方案。虽然它有冲突的概率，但对于桌面端来说，依然是跨平台实现 Deep Link 最简单、最通用的底层技术。

那为什么在文档中还时常能看到electron-fiddle://这种 URL Scheme 格式？

既然 URL Scheme（比如 electron-fiddle://）存在“容易被恶意 App 抢占”和“用户未安装时体验差”的缺点，为什么像 Electron Fiddle、VS Code (vscode://)、Slack (slack://) 这些行业标杆级别的桌面软件，依然在广泛、大量地使用这种格式呢？

这并不是因为他们“落后”，而是因为在桌面端（Windows / macOS）的生态下，URL Scheme 拥有移动端无法比拟的独特生存土壤，且它的痛点在桌面端几乎被完美规避了。

具体原因可以归纳为以下几点：

1. 桌面浏览器拥有完美的“降级引导”机制

在手机上，如果用户没装 App 却点击了 my-app://，手机浏览器会直接弹窗报错，体验极差。 但在桌面端（Chrome、Edge、Safari 等），浏览器在尝试唤醒自定义协议时，有一套非常成熟的两步走拦截机制。

当用户在网页点击 electron-fiddle:// 时，浏览器的表现通常是：

  • 如果已安装： 浏览器会弹出一个精美的系统原生提示框：“是否要在 Electron Fiddle 中打开此链接？”，点击确认即可无缝唤醒。
  • 如果未安装： 浏览器在尝试拉起协议失败后，可以通过 JavaScript 脚本在几百毫秒内迅速捕捉到这个失败，并自动引导用户去下载页。

这种桌面浏览器特有的 API 和生命周期，让自定义协议在未安装时的体验并不会变得很糟。

2. 桌面端对 Universal Link（通用链接）支持极差

在移动端，Universal Link 之所以大行其道，是因为 iOS 和 Android 系统在底层强制推行，并提供了极其丝滑的全局拦截服务。

但在桌面端：

  • Windows 系统： 根本没有成熟的、“域名双向强绑定”的 Universal Link 对应物（微软虽然有 App Links，但仅限 UWP 商店应用，对 Electron 这种主流的 Win32 应用极其不友好）。
  • macOS 系统： 虽然后期的 macOS 支持了 Universal Links，但它要求 App 必须加入 Apple Developer Program、拥有合规的开发者签名、甚至最好上架 Mac App Store 才能顺畅工作。这对于开源、免费、或者通过官网自主分发的软件（如 Electron Fiddle、各种小众工具）来说，门槛高得无法接受。

因此，URL Scheme 是目前在 Windows 和 macOS 两端唯一不需要任何服务器配置、不需要昂贵的开发者证书、100% 能够无缝落地的方案。

3. 命名空间冲突在桌面端并没那么严重

手机应用市场有几百万个 App，名字极其容易撞车。但在电脑上，用户安装的独立桌面软件数量远远少于手机。

为了防止被恶意抢占，桌面软件通常会使用非常垂直且带有强烈品牌属性的标识符。

  • 比如，别人抢占 play:// 很简单，但几乎不会有任何正经软件去抢占 electron-fiddle://、slack:// 或者 vscode-insiders://。
  • 即使在 Windows 注册表中发生了恶意覆盖，杀毒软件、系统自身的安全机制（如 Windows Defender、macOS Gatekeeper）也会对修改关键注册表 / 关联项的流氓行为进行拦截或警告。

4. 它是极佳的“跨应用互操作”工具

桌面端用户有着频繁的“多窗口、多任务协作”习惯。比如：

  • 在 浏览器 里浏览 GitHub，点击“Open in VS Code”按钮，通过 vscode:// 协议直接在本地编辑器打开代码。
  • 在 网页版 Slack 里聊天，点击“在客户端中打开”，通过 slack:// 一键切回桌面端。

这种“网页端 ─► 客户端”的无缝穿梭，在不需要复杂的域名证书配置的前提下，electron-fiddle:// 这种自定义协议就是最简单、最轻量、也最稳定的桥梁。

总结

electron-fiddle:// 这种格式之所以依然是桌面端的主流，是因为它是桌面端生态下妥协出的“最优解”。它免去了复杂的域名验证和证书签名门槛，利用桌面浏览器成熟的唤醒拦截机制，用最低的开发成本实现了近乎完美的跨端深度跳转。

之前在讨论本地 HTML 加载问题时，有提到过，将自定义本地协议注册为特权 / 标准协议；这个 Deep Links 有关系吗？

答案是：它们虽然都用了“自定义协议（Custom Protocol / Scheme）”这个词，但它们在底层完全是两码事，工作的维度一个在“外”，一个在“内”。

简单打个比方：

  • Deep Links 协议：就像你公司的对外公开客服电话。任何外面的客户（外部浏览器、其他 App）拨打这个电话，都可以直接找到你。
  • 本地特权 / 标准协议：就像你公司内部的员工对讲机 / 内线电话。只在公司大楼（Electron 渲染进程内部）使用，用于安全、快速地传递机密物资（加载本地 HTML / 图片），外面的人根本拨不通。

为了让你彻底理清，我们来做个直观的对比。

1. 深度对比：外部协议 vs 内部特权协议

是否安全受信？不受 Chromium 安全域限制（因为它压根不进入 Chromium 的沙盒路由）是。通过将其注册为“标准 / 特权（Standard / Secure）”协议，让 Chromium 把它当做像 https:// 一样安全的协议。

2. 为什么要注册为“特权 / 标准协议”？（复习内部协议）

  • 无法使用 fetch() 请求本地文件（会报跨域 CORS 错误）。
  • 无法直接在 https:// 的网页里加载 file:// 的图片。
  • 无法注册 Service Worker。

import { app, protocol, net } from 'electron';
// 必须在 app ready 之前调用
protocol.registerSchemesAsPrivileged([
  {
    scheme: 'app-assets',
    privileges: {
      supportFetchAPI: true,
      standard: true, // 像 https 一样，支持 Fetch API、CORS 跨域
      secure: true, // 标记为安全协议，避免混合内容警告
      bypassCSP: true // 绕过内容安全策略
    }
  }
]);
app.whenReady().then(() => {
  // 处理该内部协议的请求
  protocol.handle('app-assets', request => {
    const url = request.url.replace('app-assets://', '');
    // 读取本地文件并返回
    return net.fetch(`file://${__dirname}/assets/${url}`);
  });
});

一旦这样注册，Chromium 就会对 app-assets:// 大开绿灯，你的本地单页应用（SPA）就能像在服务器上运行一样流畅、无阻碍地加载。

3. 这两者绝对不能混用吗？

是的，绝对不要把它们混用。

如果你尝试用 protocol.handle('my-app', ...) 去拦截你的 Deep Link 协议（my-app://），会发生非常混乱的冲突：

  1. 内部请求被拦截：如果你的渲染进程内部尝试去 fetch('my-app://data')，它会被 Electron 的 protocol 处理器拦截，这没问题。
  2. 外部唤醒直接失效 / 死锁：但是，当用户在系统浏览器里点击 my-app:// 链接时，操作系统尝试寻找注册了该协议的 .exe。如果你的客户端内部把它当成了“标准 / 特权协议”来接管，可能会导致操作系统级的唤醒信号无法正确路由到 Electron 的 main 进程事件中，从而导致 Deep Link 机制崩溃。

最佳实践架构设计

在你的 Electron 应用中，应当清晰地各司其职：

  • 对外（Deep Link）：使用 my-app:// 协议，只用 app.setAsDefaultProtocolClient 注册，并用 app.on('second-instance')（Win）或 app.on('open-url')（Mac）去监听。
  • 对内（资源加载）：使用 app-assets:// 协议，只用 protocol.registerSchemesAsPrivileged 注册，专门用来优雅地替代 file:// 加载本地 HTML、JS、图片等。

在「构建应用菜单」部分，格式为什么都是{ role: 'xxx' }？role 是什么？Electron 有默认的枚举？

简单来说，role（角色）是 Electron 菜单（Menu）设计中极其精妙的一个概念。它代表的是一类“预设的、由系统 / 浏览器底层直接接管的原生操作”。

如果不使用 role，你想实现一个简单的“复制（Copy）”菜单，你得自己写键盘快捷键监听（Cmd / Ctrl + C）、自己获取当前焦点所在的输入框、自己读取选中的文本、自己写入剪贴板…..

但如果写了 { role: 'copy' }，Electron 会自动把这一连串复杂的底层逻辑全部封装好，甚至还会自动根据操作系统（macOS vs Windows）适配对应的快捷键和多语言翻译。

1. 为什么推荐用 role，而不是自己写 click 事件？

在构建应用菜单时，使用 role 有三个无法比拟的优势：

跨平台快捷键自动适配

不同系统的快捷键是不同的。比如“复制”：

  • 在 Windows 上是 Ctrl + C
  • 在 macOS 上是 Cmd + C

如果你写 { role: 'copy' }，Electron 会自动给 Windows 绑定 Ctrl+C，给 Mac 绑定 Command+C。如果你自己写，你必须得用 process.platform 去判断并手动写两套 accelerator。

完美的国际化（多语言）与系统原生行为

在 macOS 上，系统菜单不仅要翻译成用户系统的语言（中文显示“复制”，英文显示“Copy”），还要支持一些只有操作系统底层才能做到的原生动效（例如 Mac 上的“隐藏其他应用”、“最小化窗口”等）。

通过声明 role，Electron 会直接调用操作系统的 Cocoa API（Mac）或 Win32 API（Windows），让你的应用菜单跟原生系统软件一模一样，完全不露马脚。

2. Electron 有默认的枚举吗？

有的。在 TypeScript 中，role 的类型实际上是一个严格的联合类型（在 Electron 的定义文件中，它是 MenuItemConstructorOptions['role']）。

这些 role 大致可以分为以下几大类：

1. 编辑类（Edit），几乎每个 App 都需要

这类角色专门用来处理文本输入框的操作，由 Chromium 浏览器内核原生接管：

  • copy（复制）
  • cut（剪切）
  • delete（删除）
  • paste（粘贴）
  • redo（重做）
  • selectAll（全选）
  • undo（撤销）

2. 窗口与视图类（Window & View）

用于控制 Electron 窗口本身的行为，无需自写 BrowserWindow 的 API：

  • close（关闭当前窗口）
  • forceReload（强制刷新，忽略缓存）
  • minimize（最小化当前窗口）
  • reload（重新加载页面）
  • toggleDevTools（切换开发者工具的打开 / 关闭）
  • togglefullscreen（切换全屏）

3. macOS 专属特权类

macOS 的应用菜单有一套非常严格的规范。以下角色只在 macOS 上生效（在 Windows 上会被自动忽略，不会报错，非常安全）：

  • about（关于 App）
  • front（将所有窗口移至最前）
  • hide（隐藏当前 App，快捷键 Cmd + H）
  • hideOthers（隐藏其他 App）
  • quit（退出程序，快捷键 Cmd + Q）
  • unhide（显示全部）

大部分应用比如 VS Code 只有一个 BrowserWindow 吗？文档上，大部分都在 Main 中，只写一个 BrowserWindow 实例

在 Electron 的官方文档和各种基础教程中，确实几乎都只在主进程（Main Process）里实例化一个 new BrowserWindow()。

但在实际的工业级应用（如 VS Code、Slack、Discord）中，这个答案是：不，它们绝对不止一个 BrowserWindow。

不过，VS Code 的多窗口架构非常特殊，它和普通的“多窗口应用”有着本质的区别。要理解这一点，我们需要拆开来看 VS Code 的底层设计，以及为什么文档只教你写一个窗口。

1. 为什么文档通常只写一个 BrowserWindow？

官方文档主要是为了演示最小可行性产品（MVP）和核心 API。

在现实中，即使是单窗口应用，在许多场景下也需要创建额外的、用户“看不见”或“短暂看见”的 BrowserWindow：

  • 隐藏的计算窗口（Utility Window）： 渲染进程如果进行 CPU 密集型计算（如音视频转码、超大文件解析），会导致 UI 卡死。业界常用的做法是创建一个 show: false 的隐藏窗口，把计算丢给它，利用 Web Workers 或多进程避开主线程卡顿。
  • 独立的轻量窗口： 比如应用的“设置”面板、“关于我们”弹窗、或者像微信 / Slack 那样的“独立聊天单页”。

2. VS Code 的多窗口架构是怎样的？

当你用 VS Code 打开三个不同的项目文件夹时，你的桌面上确实会出现三个独立的窗口。在操作系统任务管理器中，你会看到三个完全独立的 BrowserWindow 实例。

然而，VS Code 在架构设计上有一个极强的原则：极力保持单个窗口（单项目）的纯粹性。

3. 为什么大部分桌面应用更倾向于“单窗口 + 虚拟 DOM 路由”？

虽然技术上可以开多个 BrowserWindow，但除了像 IDE（VS Code）、浏览器（Brave）、或者多窗口聊天软件（Slack / 微信）之外，大部分现代 Electron 应用（如 Spotify、Figma）都倾向于在单个 BrowserWindow 内部用前端路由（React / Vue / TypeScript）解决一切界面切换。

这是因为多窗口在 Electron 中有几个无法忽视的痛点：

痛点一：内存开销雪崩

每一个 BrowserWindow 背后都拖着一个完整的 Chromium 渲染引擎和一套 V8 垃圾回收机制。

  • 在 Electron 中，每多开一个空白窗口，内存就会立刻多占用 60MB - 100MB。
  • 如果你的应用只是想弹一个“确定 / 取消”的提示框，也用 BrowserWindow 去实现，用户的内存会迅速被吃光。

痛点二：跨窗口通信（IPC）极其繁琐

如果你的应用有“主窗口”和“设置窗口”：

  • 用户在“设置窗口”切换了暗黑模式。
  • “设置窗口”无法直接修改“主窗口”的 DOM。
  • 它必须发 IPC 消息给主进程：ipcRenderer.send('theme-change')。
  • 主进程收到后，再转发给主窗口：mainWindow.webContents.send('theme-change')。
  • 主窗口收到后，再更新自己的 CSS。

如果是单窗口应用，这只是前端一个全局状态（如 Redux、Pinia 或 React Context）的一行代码更新，无需跨进程通信。

总结与架构建议

在构建你的 Electron 应用时，可以遵循以下行业共识：

能用前端组件（如 Modal、Dialog）解决的，绝对不要开新 BrowserWindow。比如设置页面、确认弹窗，用 React / Vue 写在单页里，开销几乎为零。

只有在以下情况，才考虑 new BrowserWindow()：

  • 多任务并行需求： 用户需要把这个窗口拖到第二块屏幕上对照看着写代码（如 VS Code），或者需要独立的悬浮窗（如播放器迷你模式）。
  • 彻底的物理隔离： 比如运行未经信任的第三方代码、或者需要彻底独立的渲染线程进行超负荷绘图（Canvas / WebGL）。

VS Code 可以设置将多个项目通过一个创建展示（分成多个 Tab），这种情况下，也是多 BrowserWindow 架构吗？

在你说的那种情况下（VS Code 经典的 “Window: Merge All Windows”，即多项目以标签页 / Tab 形式合并在同一个窗口中展示），它的底层架构在不同操作系统上的实现机制和表现是完全不同的。

结论是：在 macOS 下，它依然是多 BrowserWindow 架构；而在 Windows / Linux 下，VS Code 并不支持这种多项目 Tab 合并，如果想在单个窗口展示多个项目，则是通过虚拟的 DOM 节点来实现的（单 BrowserWindow 架构）。

在 macOS 上，这个 Tab 栏不是 VS Code 用 HTML / CSS 画出来的，而是 macOS 系统（Cocoa 框架）原生的 NSWindow 标签页合并机制。

  • 底层对象依然是多个： 在 Electron 主进程中，VS Code 依然创建了两个完全独立的 BrowserWindow 实例，拥有各自独立的渲染进程和独立的插件宿主（Extension Host）进程。
  • 物理窗口只有一个： macOS 系统在底层检测到这两个 NSWindow 属于同一个应用，且用户设置了合并，便在系统层面将这两个窗口的绘图区域“缝合”到了同一个物理窗口的不同 Tab 里。
  • 对开发者的意义： 你的业务代码和进程架构完全不需要为了“Tab 合并”做任何特殊处理，因为在 Electron 看来，它们依然是两个完全隔离的窗口，所有的多进程优势（比如一个项目卡死不影响另一个、内存隔离）全部得以保留。

2. Windows / Linux 上的限制：不支持原生 Tab

如果你使用的是 Windows 或 Linux，你会发现 VS Code 根本没有像 macOS 那样方便的“把两个独立项目合并为两个窗口 Tab”的原生功能。

因为 Windows 和 Linux 操作系统内核并没有提供像 macOS 那样系统级、无缝、普适的 NSWindow 窗口合并 API。

如果在 Windows 下你希望在“同一个窗口”里看到多个项目，目前只有两种妥协方案：

方案 A：多工作区合并（Multi-root Workspaces）

在同一个窗口中，通过 File -> Add Folder to Workspace 把项目 B 的文件夹强行塞进项目 A 的侧边栏。

  • 架构： 这绝对是 单 BrowserWindow 架构。
  • 代价： 这两个项目的全局配置、插件运行环境全部混在了一个窗口（同一个渲染进程和插件宿主）里，失去了隔离性。

方案 B：纯前端虚拟 Tab（VS Code 目前未采用，但其他 Electron 应用常用）

有些桌面应用（如 Tabby 终端、部分第三方 Markdown 编辑器）在 Windows 上为了实现多项目 Tab，会手写一套 HTML / CSS 的 Tab 栏。

  • 架构： 单 BrowserWindow 架构。
  • 原理： 窗口内只有一个 Chromium 渲染进程。用户切换 Tab 时，前端通过 React / Vue / TypeScript 销毁上一个项目的 DOM，并加载当前项目的 DOM。
  • 痛点： 内存开销虽小，但如果一个项目卡死（比如由于超大文件高亮），整个窗口的其他 Tab 会跟着全部卡死。

最近看有些 Electron 应用集成了完整的 Chrome 来在应用内访问网站，怎么实现的？

在 Electron 应用中，想要在应用内加载、展示并控制外部网站，开发者不需要（也无法）在应用内“再打包一个完整的 Chrome”。

因为 Electron 本身就已经内置了完整的 Chromium 内核。

那些看起来在应用内做了一个“完整 Chrome 浏览器”（拥有标签页切换、地址栏、前进/后退、书签等功能）的应用（例如著名的开发工具 Rambox、Wavebox，或者一些集成了 AI 网页的客户端），在底层主要是通过 Electron 提供的以下几种“子视图 / 子页面”技术来实现的。

它们从早期的黑科技，逐步演变成了现在的官方标准。以下是它们的实现原理和具体的实现方式：

核心实现方案：WebContentsView（2026 年的现代标准）

在过去，开发者会使用 <webview> 标签或 BrowserWindow 叠加，但现在 Electron 官方主推、也是效果最完美、性能最接近原生 Chrome 的方案是 WebContentsView（在旧版本中它叫 BrowserView）。

工作原理

它的本质是在一个已有的 BrowserWindow（主窗口）上方，强行“贴”上一个或多个独立的 Chromium 视图区域。

  • 主窗口： 只负责用 React / Vue / TypeScript 渲染你 App 的外壳（比如顶部的地址栏、左侧的侧边栏、标签页按钮）。
  • WebContentsView：负责加载真正的外部网站（如 https://github.com）。它由主进程直接控制，与外壳的渲染进程完全物理隔离。

为什么这种方案像一个“完整的 Chrome”？

利用 WebContentsView 架构，你可以像写普通网页一样，用 React / Vue 轻易地手写出一个“Chrome 浏览器”：

1. 它是如何实现“多标签页（Tab）”切换的？

  • 在前端（外壳渲染进程），用户点击了“新建标签页”按钮。
  • 渲染进程通过 IPC（进程间通信）通知主进程：ipcRenderer.send('create-new-tab', url)。
  • 主进程在后台 new WebContentsView() 并加载该 URL，然后把它缓存起来。
  • 当用户切换 Tab 时，主进程只需要调用 mainWindow.contentView.removeChildView(oldView) 和 addChildView(newView)，切换过程瞬间完成，完全不消耗重新加载的流量和时间。

2. 它是如何控制网页的“前进 / 后退 / 刷新”的？

你的自定义地址栏按钮只需要通过 IPC 调用主进程中对应的 webContents API：

3. 它是如何解决“安全问题”的？

如果你直接在 Electron 的主窗口里加载外部网站，该网站可能会通过恶意脚本注入，利用你暴露的 Node.js API 读写用户的本地文件。而 WebContentsView 默认处于严格的沙盒模式（Sandbox）中，禁用了 Node.js 注入。它在 Chromium 中拥有和普通 Chrome 标签页一模一样的隔离级别，外部网站绝无可能危害到宿主电脑。

另外两种已被逐步淘汰的方案（避坑指南）

如果你在网上看一些老旧的博客，可能会提到另外两种方案，在现代 Electron 开发中不建议使用：

1. <webview> 标签

  • 用法： 直接在前端 HTML 里写 <webview src="https://github.com"></webview>。
  • 为什么淘汰： 它是基于 Chrome 早期的 Out-of-Process iFrames 机制实现的。不仅性能极差、内存消耗巨大，而且在双击、右键菜单、以及拖拽事件上存在大量无法修复的底层 Bug。Electron 官方早已不建议在生产环境使用。

2. <iframe> 标签

  • 用法： 直接写 <iframe src="https://github.com" />。
  • 为什么淘汰： 现代网站（如 GitHub、Google、淘宝等）为了防止点击劫持，几乎全部在响应头中设置了 X-Frame-Options: SAMEORIGIN 或 Content-Security-Policy。你用 iframe 加载它们，100% 会直接报拒绝连接错误，根本无法显示。

总结

你看到的“应用内集成 Chrome”，其实是用前端（React / Vue）画了一个精美的浏览器皮肤（地址栏、Tab 栏），然后用 Electron 的 WebContentsView 在皮肤下方贴上了原生的 Chromium 渲染窗口。两者的完美结合，才给用户带来了“应用内套了完整 Chrome”的极佳体验。

进阶：如何安全地“复用”用户日常的 Chrome 浏览器 Profile？

有些开发者会想：“我能不能直接把路径指向用户电脑里原生安装的 Google Chrome Profile 路径？”

强烈不建议直接读写日常使用的 Chrome Profile！

  1. 独占锁冲突： Chromium 的 User Data 目录是带锁的。如果用户的 Google Chrome 正在运行，你的客户端去强行读取同一个目录，会导致写入冲突、数据损坏，甚至直接闪退。
  2. 内核版本差异： 你的客户端内置的 Chromium 版本与用户电脑上的 Chrome 版本不一致时，会导致 Profile 数据库（如 SQL 密文格式）向上 / 向下兼容失败，甚至导致日常 Chrome 的数据损毁。
  3. 安全与解密： Chrome 存储的 Cookies 和密码在不同系统（如 macOS 的 Keychain，Windows 的 DPAPI）下是经过本地安全凭据加密的。不同 App 去解密可能会触发系统的安全警告。

业界的标准做法

  1. 多 Profile 隔离： 在你自己的应用数据目录下（如 ~/.config/alma/profiles/），为用户创建专属的虚拟 Profile 文件夹（如 profile_1，profile_2）。
  2. 导入机制： 如果要保留用户在日常浏览器里的登录状态，通常不采用“直接挂载 Chrome 目录”的做法，而是提供一个“导入 Cookie”的入口，或者在内置浏览器里让用户登录一次，之后通过 session 机制将其永久保存在上述的独立 Profile 目录下。

VS Code 中，打开多个文件，使用的是前端 Tab 切换还是多个 webContents 切换？

打开多个文件使用的是前端 Tab 切换（单渲染进程内 DOM 元素的切换与销毁），而不是多个 webContents。

VS Code 对多文件标签页的设计和渲染机制可以从以下几个维度来理解：

1. 核心架构：单渲染进程主宰 Workbench

VS Code 采用的是 Electron 架构，但它在进程管理上非常克制。

  • 一个窗口，一个渲染进程： 每一个打开的 VS Code 窗口（Window）在底层其实只对应一个 Electron 渲染进程（Renderer Process）。
  • Workbench DOM 渲染： 整个编辑器的 UI（包括侧边栏、状态栏、顶部 Tab 标签栏以及核心的文本编辑器区域）都是运行在这个单一渲染进程中的 HTML / CSS 树上。
  • DOM 切换与复用： 当你在不同的代码文件 Tab 间切换时，VS Code 只是在前端通过纯 JS / TS 逻辑（结合其自研的 Monaco Editor 实例）改变了 DOM 的挂载状态或可见性，并没有发生 Electron 级的 webContents 或进程切换。

2. 代码编辑器的复用机制（Monaco Editor）

如果每打开一个文件都创建一个新的底层编辑器实例，内存会瞬间暴涨。为此，VS Code 采取了极致的复用策略：

  • 核心编辑器实例（Monaco）是高度复用的： 通常情况下，一个 Editor Group（编辑器组）里可能只维护极少数（甚至仅一个）处于 active 状态的 Monaco Code Editor 实例。
  • 切换时只换“数据模型（TextModel）”： 当你点击不同的文件 Tab 时，前端只是把当前编辑器实例绑定的 TextModel（即包含了文件文本内容、语言、光标位置、撤销历史等状态的内存对象）替换为新文件的 TextModel。
  • 无缝过渡： 这种“只换数据不换容器”的做法，使得 Tab 切换能够在几毫秒内完成，极度流畅，同时避免了创建多个 webContents 带来的巨大开销。

3. 唯一的例外：Webview 与新版 Integrated Browser

虽然普通的代码、Markdown 等文件在同一个渲染进程内切换，但 VS Code 中确实存在使用独立沙箱或类似 webContents 机制的场景，它们通常是非纯文本文件：

  • Webview API（如 Markdown 预览、部分复杂插件 UI）： 为了隔离安全风险和防止插件 UI 搞崩主渲染进程，VS Code 里的 Webview 并不是直接写在主 DOM 里，而是放在独立的 <iframe> 中（背后可能会映射到独立的进程 / 沙箱隔离），但它们依然不是标准的窗口级 webContents 切换。
  • 集成浏览器（Integrated Browser）： VS Code 引入了基于 WebContentsView（或早期的 BrowserView）的内置浏览器。这种情况下，每个浏览器标签页背后确实对应着 Chromium 级别独立的 webContents，它像一个原生的“贴纸”一样漂浮在主窗口 DOM 的上方。

总结

对于我们日常开发中打开、切换 .ts, .json, .css 等普通代码文件，VS Code 纯粹是在同一个渲染进程里进行前端 Tab 与 DOM 的状态切换，这也是它能做到“轻量、超快响应、极省内存”的底层设计核心。

export function registerIpcHandlers() {

});

这种拆分架构带来了什么好处？

  1. 绝对的职责分离 (SoC)： index.ts 只关心“App 什么时候启动、什么时候退出、是否多开”；windowManager 只关心“窗口长宽、背色、怎么加载 HTML”；ipc 只关心“渲染进程发来消息，我该如何返回数据”。
  2. 避免循环依赖 (Circular Dependency)： 这是 Electron 拆分最容易踩的坑（比如 index.ts 引用了 ipc.ts，而 ipc.ts 内部为了给窗口发消息，又去引用 index.ts 里的 mainWindow 变量）。 通过引入单例模式（Singleton）的 WindowManager，其他任何子模块（如 IPC、自动更新）只要想获取主窗口，直接通过 WindowManager.getInstance().getMainWindow() 即可拿到，完美避开了循环引用。
  3. 极易编写单元测试： 当业务逻辑被拆分到 services 下的纯类（Class）或纯函数中后，你可以非常容易地对这些不依赖 GUI 的 Node.js 代码编写单元测试。

NavigationHistory 与 History API 的关系

你提到的 NavigationHistory（严格来说，目前在现代 Web 标准中正式的名称叫 Navigation API）是前端路由技术的一次跨时代演进。

简单来说：Navigation API 是为了彻底替换和解决老旧的 History API (window.history) 的痛点而诞生的一套全新现代 Web 标准。

它们的关系就像是原生的 fetch() 替换老旧的 XMLHttpRequest 或者是 async / await替换复杂的Promise.then 链。

下面为你梳理它们底层的核心区别，以及为什么单页应用（SPA）开发者会如此期待这次更迭。

1. 为什么我们需要替代 History API？（痛点）

过去十几年里，React Router、Vue Router 等现代前端路由全部基于 window.history（pushState / replaceState）构建。但这个 API 设计于早期的 Web 时代，对于现在的单页应用（SPA）极其不友好，存在以下致命的三个痛点：

痛点一：无法拦截和监听 pushState

这是最让人痛苦的地方。当路由发生变化时，History API 只提供了一个 popstate 事件。

  • popstate 只在用户点击浏览器的“前进/后退”按钮时触发。
  • 当你自己在代码里调用 history.pushState() 或 history.replaceState() 切换路由时，没有任何原生事件会触发！
  • 后果： 所有的前端路由库（如 React Router）都必须通过“重写 / 猴子补丁 (Monkey Patching)”原生 pushState 方法来强行监听路由变化，实现非常魔改。

痛点二：无法取消（Cancel）路由过渡

如果你正在写一个复杂的表单，用户不小心点了左上角的后退按钮，你想弹窗提示：“表单尚未保存，确定离开吗？”

  • 在旧的 History API 中，当 popstate 触发时，浏览器的地址栏已经改变了，历史记录也已经后退了。你无法通过 event.preventDefault() 阻止这次发生过的行为，只能通过再次调用 pushState 强行把历史记录“顶回来”，这会导致历史栈变得一团糟。

痛点三：对异步过渡和无障碍（Accessibility）一无所知

当点击 Tab 切换路由时，前端往往需要去异步 fetch 新页面的数据，数据到了再渲染 DOM。

  • 旧的 History API 根本不知道你的应用正在执行异步操作。
  • 这导致浏览器顶部的加载进度条（Loading Indicator）不会转动，屏幕阅读器也无法感知到页面正在发生宏观跳转，对无障碍极不友好。

2. 现代的 Navigation API 是如何颠覆的？

全新的 Navigation API (window.navigation) 是一套由 Chrome 主导、专门为单页应用（SPA）量身定制的历史与导航控制中心。

优势一：全局唯一的 navigate 集中监听器

无论是用户点后退按钮、点击带 href 的常规 <a> 标签，还是你在代码里触发跳转，全部会触发统一的 navigate 事件：

1
2
3

window.navigation.addEventListener('navigate', event => {
  console.log('检测到全局任何形式的路由跳转目标：', event.destination.url);
});

优势二：支持真正完美的“中途拦截与取消”

想要做“表单未保存拦截”？现在只需要一行轻量级的代码：

1
2
3
4
5
6
7

window.navigation.addEventListener('navigate', event => {
  if (isFormDirty) {
    // 真正地阻止导航发生！地址栏不会变，历史记录栈保持原样
    event.preventDefault();
    showSaveConfirmModal();
  }
});

优势三：完美原生支持异步过渡（Transition）

当你要异步加载新页面数据时，你可以把 Promise 传给 event.intercept()。 浏览器会瞬间感知到这是一个异步导航，系统会自动让浏览器的原生 Loading 转圈，并在 resolve 时通知屏幕阅读器更新，让 SPA 体验无限趋近于传统多页应用。

window.navigation.addEventListener('navigate', event => {
  // 拦截当前导航，改为单页内部接管
  event.intercept({
    async handler() {
      // 1. 异步请求新页面的数据
      const data = await fetchPageData(event.destination.url);
      // 2. 更新 React / Vue 的状态来渲染新组件
      renderNewDOM(data);
    }
  });
});

3. 它们的数据模型也有本质不同

  • History API 的状态（history.state）：是完全盲目的。你只知道当前页面有一个 state 对象，但你根本不知道历史记录栈里前一个页面是什么，后一个页面是什么，整个栈对你来说是个黑盒。
  • Navigation API 的状态（navigation.entries()）：将历史记录栈透明化为一个直观的数组。你可以清晰地通过 navigation.currentEntry 获取当前路由节点，或者通过数组查看到用户在这一页之前和之后的所有历史快照，甚至能精确定位每个节点的 ID。

总结与工程现状

  1. 概念关系： History API 是属于 Web 1.0 / 2.0 时代的遗留产物；Navigation API 是 Web 3.0 时代为单页应用（SPA）架构重构的原生核心底座。
  2. 当前现状： 截止到 2026 年，Navigation API 已在 Chromium 系浏览器（Chrome、Edge、Opera 以及我们常用的 Electron 内核）中得到了极其稳定的原生支持，各大主流前端路由库（如 React Router 现代版）的底层也逐步增加了对其的无缝适配。如果在不受支持的浏览器（如老版本 Safari）上，目前社区也提供了非常成熟的 Polyfill 垫片进行平滑降级。

Electron 文档中的 NavigationHistory 就是 Navigation API？

不完全是一回事，这是两个处在不同层级、由不同角色实现的 API。

简单来说：

  • Navigation API（window.navigation）是 Chromium 浏览器环境（前端渲染进程）提供给网页 JavaScript 的标准 Web API。
  • Electron NavigationHistory 是 Electron 框架（主进程）暴露给 Node.js 端的 API，用来让你的主进程代码可以直接操控某个窗口（webContents）的底层历史记录栈。

虽然它们都围绕“浏览历史”展开，但它们的工作位置和应用场景完全不同：

对比：Electron 的 NavigationHistory vs Web 的 Navigation API

1. 运行环境与访问方式

  • Web Navigation API：运行在渲染进程（前端）。你在前端组件（React / Vue）或网页脚本里直接通过全局变量 window.navigation 访问。
  • Electron NavigationHistory：运行在主进程（后端 / Node 端）。你需要通过窗口的 webContents 实例来调用：
1
2

// 主进程
const { navigationHistory } = mainWindow.webContents;

2. 它们的核心职责不同

  • Web Navigation API 专注于“拦截”与“应用内路由控制”。它的强项是在网页跳转发生前，让前端 JS 能够拦截请求（event.intercept），实现无刷新的单页应用（SPA）无缝跳转。
  • Electron NavigationHistory 专注于“窗口级别的历史管理与克隆”。它是以一种“造物主”的视角，从主进程去读取、控制和修改 Chromium 实例的历史栈。

Electron 的 NavigationHistory 能做哪些前端做不到的事？

因为 NavigationHistory 是主进程的特权 API，它提供了一些纯前端（包括 Navigation API）绝对无法实现的底层能力：

功能一：完美的“撤销关闭标签页 / 克隆标签页”（History Restore）

在开发类似于浏览器（如你之前提到的 Alma 内置浏览器）的客户端时，如果用户关闭了一个 Tab，或者你想“复制”当前窗口到新窗口，前端的 window.history 或 window.navigation 是没办法完整把整个历史队列导出来并注入给新窗口的。

而在主进程中，你可以用 NavigationHistory 轻松做到克隆和恢复：

 2

10

// 1. 获取窗口 A 的完整历史快照
const activeIndex = windowA.webContents.navigationHistory.getActiveIndex();
const entries = windowA.webContents.navigationHistory.getAllEntries();
// 2. 将这段历史完美克隆并注入到新建的窗口 B 中
const windowB = new BrowserWindow();
windowB.webContents.navigationHistory.restore({
  index: activeIndex,
  entries: entries
});
// 此时 windowB 不仅能显示当前页，点击它的“后退”和“前进”也完全有效！

功能二：主进程直控的自由跳转（Jump Anywhere）

不需要向前端发 IPC 通信，主进程可以直接跨越式导航：

if (navigationHistory.canGoToOffset(-3)) {
  navigationHistory.goToOffset(-3); // 直接后退 3 页
}

总结

可以把它们看作是一枚硬币的两面：

  • 如果你想解决前端网页内点击链接如何拦截、如何做 SPA 异步过渡、如何优雅地提示“表单未保存”，请在前端代码里使用现代 Web 的 Navigation API。
  • 如果你在主进程编写客户端外壳逻辑，需要做类似“自定义标签页管理”、“撤销关闭标签页”、“多窗口同步历史记录”或开发自定义的浏览器外壳工具，请使用 Electron 的 NavigationHistory。
